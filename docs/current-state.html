quant-engine — Current State & Maintainer's Guide

A deterministic, information-firewalled crypto-perp quant trading platform in Rust.

Entry point for developers maintaining the repo and for users evaluating it for investment. Everything below was verified against the codebase, not assumed. Generated 2026-07-03, refreshed 2026-07-18 · reflects Phase 2 (through G2), Phase PreP3 complete, and the now-complete QE-430..454 program (quant-methodology hardening + the GP indicator-evolution subsystem).

1 · TL;DR

What this is. A 26-crate Cargo workspace (plus a web/ React SPA) that builds a full pipeline for discovering, validating, and (eventually) live-trading systematic strategies on USDⓈ-M perpetual futures (dev universe: BTCUSDT, ETHUSDT). The design is organised around two hard rules enforced in CI: everything is deterministic & reproducible from a recorded vintage hash, and the strategy search can never see live or portfolio outcomes (the "information firewall").

Where it stands. 80 tickets complete. Phase 0 (foundations), Phase 1 (training → backtest validation, through the G1 gate), Phase 2 (runtime → paper/sim), through the G2 shadow/dry-run gate, and Phase PreP3 (admin UI for training & backtesting) are fully implemented and tested. The full runtime loop and the admin platform — CLI jobs, an axum server with Google-OAuth auth, and a React SPA — all exist and pass tests. Only Phase 3 (live capital, attribution & ops, through the independent G3 go/no-go) remains, plus a short list of PreP3 hardening follow-ups (§8).

The one thing to understand. The pipeline is now genuinely runnable: qe-cli exposes train (real MAP-Elites search → ensemble → G1 → sealed vintage), backtest (a sealed vintage over a window → deterministic result), and ingest; a second binary qe-server triggers/supervises those jobs and serves an authenticated SPA to trigger, monitor, and review runs in a browser. Everything is still paper/offline: there is no live order submission anywhere (the order path terminates in a kill-gated simulator; live HTTP/wss sit behind a default-off http feature).

What's new since PreP3. The completed QE-430..454 program landed the 20 Max-Dama quant-methodology hardening recommendations (QE-430..449) and a new GP indicator-evolution subsystem (QE-451/452/453/454): an offline, default-off symbolic-regression stage (qe evolve) that illuminates a pool of evolved indicator formulas, and a governed production-seal path (RBAC + a server-authoritative seal_allowed predicate + a tamper-evident audit log) that decides whether a pool may ever reach production. It is default-off and golden-safe — no CATALOGUE_VERSION/VINTAGE_FORMAT_VERSION moved.

The GP engine + governance are shipped and armed, but the live evolve→production-seal path is intentionally FAIL-CLOSED. Everything is built, tested, and merged, but a live-evolve-produced pool cannot seal to production today: the seal predicate's per-formula gate_evidence hard-blocks fire until that evidence is wired into the evolve pipeline (a deferred follow-up, §8). This is the safe direction by design, not a defect — sandbox/research illumination and inspection work end-to-end now.

2 · What it does today

Two executables ship today: qe-cli (the deterministic pipeline jobs) and qe-server (the admin backend that supervises those jobs and serves the SPA).

CommandWhat it actually does today
qe-cli version Prints quant-engine 0.1.0.
qe-cli train
  --config <file>		 Runs the real training pipeline: MAP-Elites/WFO search → discrete-DE ensemble → DSR/SPA validation → the G1 gate → seals a content-hashed vintage under the artifacts dir. Emits JSON-line progress (generation, MAP-Elites archive coverage, CV folds, best-so-far, G1 pass/fail). Deterministic for a fixed seed (same seed ⇒ same vintage id).
qe-cli backtest
  --vintage <id> --start --end
  --resolution … --run-dir …		 Loads + verifies a sealed vintage, builds decision bars via the shared feature pipeline, runs each chromosome through the backtester, weight-aggregates to ensemble returns, and writes a deterministic result.json (CAGR/Sharpe/Sortino/max-DD/win-rate/profit-factor, equity & drawdown curves, monthly heatmap, per-trade rows). JSON-line progress ends with done.
qe-cli ingest
  --config --start --end --resolution		 Populates the LMDB market store from an injectable historical source (real Binance decoders stay behind the default-off http feature). Also exposes the read-only coverage query the admin UI's Market-data view uses.
qe-cli evolve
  --run-dir … --json		 Runs the offline GP indicator search (QE-451 engine: Expr-tree illumination → GP-aware deflation + tradability gates → freeze K≤16) and seals a content-hashed qe-formula-pool artefact — K≤16 canonical S-expressions + a deflation summary + review lineage — under data/artifacts/{research|…}/pools/. Emits JSON-line progress; Done emits pool: and never vintage:. Default-off / research-first; the pool is offline data only (the runtime never loads it).
qe-server axum + tokio backend (a second composition root — reuses the training/shared crates, never qe-runtime). Google-OAuth login gated by QE_ADMIN_ALLOWED_EMAILS + signed cookie; a file-based run store (data/runs/<id>/…) that spawns & supervises qe-cli backtest/train subprocesses; /api for runs, vintages, and market-data coverage; and serves the built React SPA at / (same origin).

The React SPA (web/, Vite + TypeScript, the ported Claude Design system) gives an authenticated browser UI: Login, Backtests (list / new / result with live progress + the full result contract), a Training monitor (generations, archive-coverage grid, CV folds, best-so-far → G1 verdict → link to the sealed vintage's backtest), and read-only Market-data coverage. Trade/Risk surfaces are present but disabled (Phase 3). A new evolve/ area (NewCampaign / CampaignMonitor / FormulaSexpr / PoolReview / PoolBrowser) launches and monitors GP campaigns and drives the pool-governance gate — it honestly surfaces the server's fail-closed reality (a production seal is refused, never faked).

Two capabilities landed with the QE-430..454 program. First, a GP indicator-evolution subsystem (offline, default-off): the qe evolve job illuminates a pool of evolved indicator formulas and seals it as a qe-formula-pool artefact, separate from the training vintage. Second, a governed production-seal path: per-request RBAC, a tamper-evident audit log, dual sign-off, and a server-authoritative seal_allowed predicate (8 hard-blocks + 3 structural barriers) decide whether a pool may reach production. The engine and governance are shipped and armed — but, as noted above, the live evolve→production-seal path is intentionally fail-closed until per-formula gate_evidence is wired into the evolve pipeline (§8).

The full runtime loop exists as tested libraries (Phase 2, through G2)

Beyond the runnable jobs above, the entire live/paper runtime is built and tested but is exercised through cargo test, not yet behind an operator command:

  • Venue-aware REST client + wss Market/Realtime/user-data streams (behind the default-off http feature)
  • Live kline source (REST-prime + wss-stitch), factor join, mark-EMA loop; the shared evaluator session (replay & live, no state copy), deterministic cold-start bootstrap + in-process cutover with restart parity
  • Circuit breakers, position netting, the Hedge Planner, pre-trade risk, the out-of-band kill switch, the venue adapter + position keeper + paper simulator, planner↔edge transport, vintage load + rollover
  • Real-time reconciliation divergence alarm and the G2 shadow/dry-run gate: would-be orders reconciled vs the simulator, nothing submitted

Test baseline (after QE-430..454): 1046 Rust tests (+2 ignored) across the workspace, plus 75 frontend tests (Vitest/RTL), all green; firewall + decoupling guards green. There is still no live order submission anywhere — the order path terminates in a kill-gated paper simulator.

3 · How to run it

Prerequisites

The Rust toolchain pinned by rust-toolchain.toml (cargo on PATH) for the CLI/server; Node (for building the SPA). No database, no API key, no network needed to run the offline jobs against the committed sample store.

Run the pipeline (CLI)

# train the real pipeline → seals a content-hashed vintage (deterministic per seed)
cargo run -p qe-cli -- train --config config.example.toml
#   → JSON-line progress (generations → G1) → data/artifacts/vintages/<id>/

# backtest a sealed vintage over a window → deterministic result.json
cargo run -p qe-cli -- backtest --vintage <id> --start 2021-01-01 --end 2024-12-31 \
    --resolution 1h --run-dir /tmp/run --json

# populate the LMDB market store (real network decoders behind the default-off `http` feature)
cargo run -p qe-cli -- ingest --config config.example.toml --start … --end … --resolution 1h

# run the offline GP indicator search → seals a K≤16 formula pool (default-off; Done emits pool:, never vintage:)
cargo run -p qe-cli -- evolve --run-dir /tmp/evolve --json
#   → data/artifacts/{research|…}/pools/<id>  (canonical S-exprs + deflation summary + review lineage)

Run the admin UI (server + SPA)

# 1) build the SPA
cd web && npm ci && npm run build      # → web/dist

# 2) run the server (serves web/dist at / and the API at /api)
QE_SERVER_STATIC_DIR=web/dist \
QE_ADMIN_ALLOWED_EMAILS=you@example.com \
QE_GOOGLE_CLIENT_ID=… QE_GOOGLE_CLIENT_SECRET=… QE_GOOGLE_REDIRECT_URI=… QE_SESSION_SECRET=… \
    cargo run -p qe-server
#   → sign in with Google (allowlisted) → trigger/monitor/review backtest & training runs

What you provide

Only a config file. config.example.toml works as-is. Every field is overridable by a QE_-prefixed environment variable (nested with __):

QE_DETERMINISM__SEED=7 cargo run -p qe-cli -- train --config config.example.toml

Config shape (TOML): profile, instruments / [[universe]] (with listed/delisted dates), [bars] (base + reconstructed resolutions), [storage] directories (relative, volume-friendly), [determinism].seed.

4 · Architecture

The workspace is 26 crates (+ a web/ SPA). The organising idea is a shared spine (qe-domain + qe-signal) that both the offline training world and the online runtime world depend on — and nothing else crosses between them. That separation is not a convention; it is checked by tests (see §6).

How the dependencies actually wire (verified)

Crate dependency map. Training (left) feeds the shared spine; runtime (right) reads it and loads the sealed vintage read-only. qe-cli is the only crate that touches both sides.
  • qe-signal → qe-domain only. It is the narrowest crate and the spine of everything.
  • qe-runtime → {domain, risk, signal, storage, venue, vintage, determinism} — never wfo or ensemble.
  • qe-vintage → {signal, risk, determinism}. Because the strategy genome lives in signal (relocated in QE-207), a vintage can carry chromosomes to the runtime without dragging in the training crates — that is what keeps the two sides decoupled.
  • qe-cli depends on both worlds; it is the only place they meet, and it is the intended place.

The GP indicator-evolution subsystem QE-451/452/453/454

Layered on the training side and its own server/SPA surface, all default-off and firewall-clean:

Engine & artefact

  • GP engine — qe-wfo/src/gp/: an Expr/Kernel FIR tree interpreter (Decimal-only, batch == streaming by construction), an Elite<ExprTree> MAP-Elites archive, GP-aware deflation (uncensored PBO primary, DSR necessary-not-sufficient) + tradability gates (IC / cost-stress / turnover / capacity) + a K≤16 freeze.
  • qe-formula-pool — a new pure leaf crate, separate from qe-vintage: K≤16 canonical S-expr strings + a deflation summary + review lineage. It reuses the Vintage seal/verify/load SHA-256 discipline verbatim (load never returns an unverified pool), under a separate directory root. The runtime never loads a pool.

Server surface & SPA

  • Read routes — GET /api/formula-pools[/{id}], GET /api/runs/{id}/archive.
  • Governance routes — POST /api/formula-pools/{id}/{approve,reject,revoke,seal}, POST /api/runs/{id}/halt, GET /api/audit.
  • SPA — web/src/app/evolve/: NewCampaign · CampaignMonitor · FormulaSexpr · PoolReview · PoolBrowser.

Ops-safety / governance layer QE-454

The seal path is a real safety control, enforced server-side:

  • Per-request RBAC — require_role resolved per request from env allowlists (QE_ROLE_OPERATORS/APPROVERS/ADMINS) keyed by the session-derived email, never from cookie/body.
  • Tamper-evident audit log — hash-chained (entry_hash = SHA256(canonical‖prev)) + HMAC under QE_AUDIT_SIGNING_KEY; GET /api/audit verifies the chain; fail-closed if the key is unset.
  • GovernanceRecord — a separate content-addressed leaf outside VintageContent (vintage byte-identity preserved); dual sign-off = two distinct approvers ≠ launcher.
  • Server-authoritative seal_allowed — the single choke point: an AND over {hash-verified pool, audit replay, compiled const} requiring the 8 §13.5 hard-blocks + mode == production + not-revoked + dual sign-off; any failure ⇒ 409 + named blockers + an appended Reject audit entry.
  • 3 structural barriers — the compiled DEFLATION_BASIS_VERSION prereq bitset (barrier 1, launch-time 400), a research pool root the production VintageRepository never scans (barrier 2), and a fail-closed assert_production_eligible keyed on the hashed mode (barrier 3).
  • Run supervision — a ≈24h deadline (reuses the QE-407 kill path) + a dedicated evolve semaphore (QE_SERVER_MAX_EVOLVE_CONCURRENCY) serialising campaigns without starving backtests.

5 · Data flow (end-to-end intent)

The spec numbers the pipeline ①→⑦ on the training side and ③/④ on the runtime side. The vintage is the artefact that bridges them.

The vintage artefact (⑦) bridges training and runtime: it passes the G1 gate, then the runtime bootstraps from it and runs the live loop toward the G2 dry-run gate. No orders are sent before G2.

The GP evolve path — a second, parallel lifecycle

The evolve flow adds a two-lifecycle model that is deliberately distinct from train→vintage. An ephemeral evolve RUN (illuminate → deflation/gates → freeze) produces a sealed pool; that pool then has its own durable POOL lifecycle, tracked and governed separately from the run:

Draft → Approved → Sealed   (+ Rejected / Revoked)

Guarded transitions reject every illegal edge; the lifecycle state is persisted alongside the pool artefact (under <data_dir>/governance), not in the run. Crucially, a sealed pool does NOT auto-mint a vintage — the evolve lifecycle and the train→vintage lifecycle never cross that way. Done from an evolve run emits pool:, never vintage:.

6 · The two firewalls (why this codebase is shaped the way it is)

Two CI-enforced invariants drive nearly every architectural decision. A maintainer must understand both before moving code between crates.

Both sides may depend into the shared spine (green); neither may depend on the other directly (red). That is why the genome lives in qe-signal and the vintage is plain data.

① Information firewall QE-132

Test: crates/architecture/tests/firewall.rs — directional.

The evolutionary search must not reach into portfolio or live concerns. It asserts what qe-wfo and qe-ensemble are allowed to depend on (not the reverse): each is forbidden from reaching runtime, venue, or each other. This stops the search from ever "seeing" outcomes it must be blind to.

② Train/live decoupling QE-001

Test: crates/cli/tests/dependency_topology.rs — bidirectional.

qe-runtime must not transitively reach qe-wfo/qe-ensemble, and they must not reach qe-runtime. "The only code shared between the training and runtime sides crosses through signal/domain." This is why the genome lives in signal and the vintage is plain data.

Practical rule for contributors. If you need to share something between the training and runtime sides, it goes in qe-signal or qe-domain — nowhere else. If you find yourself wanting qe-runtime to depend on qe-wfo, stop: the decoupling test will fail, and the right move is to push the shared type down into the spine (exactly what QE-207 did with the genome).
The GP subsystem respects both firewalls. The pool/governance boundary is firewall-clean: the evolved formulas' deflation statistics cross into the seal_allowed predicate as plain DATA (no crate edge), and qe-runtime/qe-venue never depend on qe-formula-pool — the runtime never loads a pool. The search⟂portfolio firewall (QE-132) is unchanged: qe-wfo/src/gp/ still cannot reach runtime or venue.
