quant-engine — Current State & Maintainer's Guide

A deterministic, information-firewalled crypto-perp quant trading platform in Rust.

Entry point for developers maintaining the repo and for users evaluating it for investment. Everything below was verified against the codebase, not assumed. Generated 2026-07-02 · branch docs/refresh-current-state-p2.

1 · TL;DR

What this is. A 20-crate Cargo workspace that builds a full pipeline for discovering, validating, and (eventually) live-trading systematic strategies on USDⓈ-M perpetual futures (dev universe: BTCUSDT, ETHUSDT). The design is organised around two hard rules enforced in CI: everything is deterministic & reproducible from a recorded vintage hash, and the strategy search can never see live or portfolio outcomes (the "information firewall").

Where it stands. 69 / ~75 tickets complete. Phase 0 (foundations), Phase 1 (training → backtest validation, through the G1 gate), and now Phase 2 (runtime → paper/sim), through the G2 shadow/dry-run gate, are fully implemented and tested. The full runtime loop — live data plumbing, shared evaluator, deterministic cold-start bootstrap, in-process cutover, circuit breakers, netting, the Hedge Planner, pre-trade risk, the out-of-band kill switch, the venue adapter + simulator, real-time reconciliation, and the G2 shadow gate — all exist and pass tests. Only Phase 3 (live capital, attribution & ops, through the independent G3 go/no-go) remains.

The one thing to understand. The sophisticated machinery lives as libraries with deep test coverage. The single shipped binary (qe-cli) currently exposes only version and a train command that writes a provenance manifest — it does not yet run the search/backtest or the live loop end-to-end. Connecting the libraries into runnable pipelines is itself the next body of work (see §8).

2 · What it does today

There is exactly one executable, qe-cli, with two commands:

CommandWhat it actually does today
qe-cli version Prints quant-engine 0.1.0.
qe-cli train
  --config <file>		 Loads a TOML config → resolves the point-in-time universe (respects listing/delisting dates, no survivorship bias) → creates the storage directories → computes a deterministic content-hash lineage → writes a vintage manifest (<artifacts_dir>/vintages/<id>/manifest.json) recording the id, profile, config/seed lineage, and universe roster.
Important — train is a provenance skeleton, not yet a trainer. It does not ingest market data, run indicators, evolve strategies, or build an ensemble. The manifest's input_snapshot_id is intentionally empty. The real training stack (ingest, the indicator catalogue, the walk-forward search, the ensemble, the G1 validation gate) is built and passing tests but is not yet wired behind this command.

Capabilities that exist as tested libraries (not yet behind a command)

Training side (Phase 1 — complete)

  • Binance public-dumps downloader + REST month-to-date backfill, with data-integrity reconciliation
  • Fusion / normalisation → LMDB market store
  • Multi-resolution bar reconstruction (5m → 30m/4h), batch & streaming-identical
  • ~20+ quantised indicator catalogue → feature vectors → synthetic store
  • Execution-friction & funding model (VIP0 fees, real funding series, slippage)
  • QD/MAP-Elites strategy search, Thompson parent selection, purged/embargoed CV, robustness gates
  • Discrete-DE ensemble with correlation/capacity/regime constraints → calibrated vintage artefact
  • Stress / worst-case-loss, statistical robustness (DSR/SPA), and the G1 holdout gate

Runtime side (Phase 2 — complete, through G2)

  • Venue-aware REST client: weighted rate-limit back-off, retry budget, ephemeral closed-window cache
  • wss Market- & Realtime-tier streams + user-data stream + tier-partitioned connection registry
  • Live kline source (REST-prime + wss-stitch), live factor join, mark-EMA tick loop
  • Evaluator session (one object, replay & live, no state copy) + deterministic cold-start bootstrap + in-process cutover with restart parity
  • Circuit-breaker layer, position netting, the Hedge Planner (absolute target model), and pre-trade risk (clamp/reject governor)
  • Venue adapter + position keeper + paper simulator, an out-of-band kill switch at the adapter, gRPC-style planner↔edge transport, and read-only vintage load + rollover
  • Real-time reconciliation divergence alarm (debounced, can trip the kill) and the G2 shadow / dry-run gate: would-be orders reconciled vs the simulator, nothing submitted

These are exercised through cargo test (578 tests + 1 ignored, 57 suites, all green). There is still no live order submission anywhere — the order path terminates in a kill-gated paper simulator, and the live HTTP/wss transports sit behind a default-off http feature flag.

3 · How to run it

Prerequisites

The Rust toolchain pinned by rust-toolchain.toml (cargo on PATH). No database, no API key, no network needed for what runs today.

Run the binary

# print version (the bare invocation also prints version)
cargo run -p qe-cli -- version

# run train against the bundled example config
cargo run -p qe-cli -- train --config config.example.toml
#   → produced vintage <64-hex-id> → data/artifacts/vintages/<id>/manifest.json

# inspect the output
cat data/artifacts/vintages/*/manifest.json

What you provide

Only a config file. config.example.toml works as-is. Every field is overridable by a QE_-prefixed environment variable (nested with __):

QE_DETERMINISM__SEED=7 cargo run -p qe-cli -- train --config config.example.toml

Config shape (TOML): profile, instruments / [[universe]] (with listed/delisted dates), [bars] (base + reconstructed resolutions), [storage] directories (relative, volume-friendly), [determinism].seed.

4 · Architecture

The workspace is 20 crates. The organising idea is a shared spine (qe-domain + qe-signal) that both the offline training world and the online runtime world depend on — and nothing else crosses between them. That separation is not a convention; it is checked by tests (see §6).

Crate map by layer

LayerCratesRole
Foundation
(cross-cutting)		 domain · error · config · determinism · clock · telemetry · risk Types (bars, instruments, time), error taxonomy, layered TOML config, the reproducibility/lineage harness, time-sync guard, structured logging, and the risk-limit / kill-switch contract.
Storage storage LMDB market-data & synthetic-feature stores (no database on the runtime hot path).
Shared spine domain · signal signal holds bar reconstruction, the indicator catalogue, feature assembly, regime labelling, and the strategy genome — the one piece of strategy logic both sides must run identically.
Training ingest · wfo · ensemble · vintage Ingest/fusion; the walk-forward QD search (wfo); the discrete-DE portfolio builder (ensemble); the sealed, content-hashed vintage artefact that is the hand-off to runtime.
Runtime venue · runtime venue = REST/wss connectivity; runtime = live kline source, factor join, evaluator session, bootstrap pipeline (and the future risk/order loop).
Validation & reporting validation · gate · report Statistical robustness suite, the phased quality gates (G1…), and validation reports.
Guards & root architecture · cli architecture hosts the information-firewall CI guard; cli is the composition root (the only crate that may touch both training and runtime).

How the dependencies actually wire (verified)

Crate dependency map. Training (left) feeds the shared spine; runtime (right) reads it and loads the sealed vintage read-only. qe-cli is the only crate that touches both sides.
  • qe-signal → qe-domain only. It is the narrowest crate and the spine of everything.
  • qe-runtime → {domain, risk, signal, storage, venue, vintage, determinism} — never wfo or ensemble.
  • qe-vintage → {signal, risk, determinism}. Because the strategy genome lives in signal (relocated in QE-207), a vintage can carry chromosomes to the runtime without dragging in the training crates — that is what keeps the two sides decoupled.
  • qe-cli depends on both worlds; it is the only place they meet, and it is the intended place.

5 · Data flow (end-to-end intent)

The spec numbers the pipeline ①→⑦ on the training side and ③/④ on the runtime side. The vintage is the artefact that bridges them.

The vintage artefact (⑦) bridges training and runtime: it passes the G1 gate, then the runtime bootstraps from it and runs the live loop toward the G2 dry-run gate. No orders are sent before G2.

6 · The two firewalls (why this codebase is shaped the way it is)

Two CI-enforced invariants drive nearly every architectural decision. A maintainer must understand both before moving code between crates.

Both sides may depend into the shared spine (green); neither may depend on the other directly (red). That is why the genome lives in qe-signal and the vintage is plain data.

① Information firewall QE-132

Test: crates/architecture/tests/firewall.rs — directional.

The evolutionary search must not reach into portfolio or live concerns. It asserts what qe-wfo and qe-ensemble are allowed to depend on (not the reverse): each is forbidden from reaching runtime, venue, or each other. This stops the search from ever "seeing" outcomes it must be blind to.

② Train/live decoupling QE-001

Test: crates/cli/tests/dependency_topology.rs — bidirectional.

qe-runtime must not transitively reach qe-wfo/qe-ensemble, and they must not reach qe-runtime. "The only code shared between the training and runtime sides crosses through signal/domain." This is why the genome lives in signal and the vintage is plain data.

Practical rule for contributors. If you need to share something between the training and runtime sides, it goes in qe-signal or qe-domain — nowhere else. If you find yourself wanting qe-runtime to depend on qe-wfo, stop: the decoupling test will fail, and the right move is to push the shared type down into the spine (exactly what QE-207 did with the genome).

Cross-cutting gaps a maintainer should pick up

  • Wire the libraries into runnable commands. The largest gap between perception and reality: qe-cli train writes a manifest but does not yet run ingest → search → ensemble → G1. A real train / backtest / bootstrap subcommand surface is needed to make the (already tested) pipeline observable end-to-end.
  • Add the missing Dockerfile the README promises (deployment-agnostic packaging, QE-013's intent), or correct the README.
  • Deferred analytics: Parquet export + DuckDB (QE-135) and the signal viewer (QE-136) for inspecting features/strategies.
  • Bookkeeping convention (now established): docs/mds/work.md is a transient scratchpad for the PR under review only; every merged ticket's approved review record is archived to docs/mds/reviewed/<ticket>.md (all 69 completed tickets, QE-001→QE-222, have records there) and work.md is cleared. Because main is branch-protected, archive commits ride the next ticket's PR through CI.

For someone evaluating this to invest

This system cannot trade with real capital today. Phase 2 is now complete through the G2 shadow/dry-run gate — the full runtime loop runs against live data and produces would-be orders reconciled against a simulator — but the order path still terminates in a paper simulator: there is no live broker submission, and the training pipeline is not yet runnable end-to-end from the CLI. Phase 3 must still complete through the independent G3 go/no-go with a staged, capped capital ramp before any real capital is involved.

What is strong and verifiable: a disciplined, deterministic, information-firewalled design with deep test coverage, a friction-true backtest (fees + real funding + slippage so no strategy is ranked on returns that won't exist live), and statistical over-fit gates (deflated Sharpe, SPA) baked into the validation milestone. The engineering rigour is real; the product is mid-build.
