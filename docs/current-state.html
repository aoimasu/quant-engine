quant-engine — Current State & Maintainer's Guide

A deterministic, information-firewalled crypto-perp quant trading platform in Rust.

Entry point for developers maintaining the repo and for users evaluating it for investment. Everything below was verified against the codebase, not assumed. Generated 2026-07-03 · reflects Phase 2 (through G2) and Phase PreP3 complete.

1 · TL;DR

What this is. A 21-crate Cargo workspace (plus a web/ React SPA) that builds a full pipeline for discovering, validating, and (eventually) live-trading systematic strategies on USDⓈ-M perpetual futures (dev universe: BTCUSDT, ETHUSDT). The design is organised around two hard rules enforced in CI: everything is deterministic & reproducible from a recorded vintage hash, and the strategy search can never see live or portfolio outcomes (the "information firewall").

Where it stands. 80 tickets complete. Phase 0 (foundations), Phase 1 (training → backtest validation, through the G1 gate), Phase 2 (runtime → paper/sim), through the G2 shadow/dry-run gate, and Phase PreP3 (admin UI for training & backtesting) are fully implemented and tested. The full runtime loop and the admin platform — CLI jobs, an axum server with Google-OAuth auth, and a React SPA — all exist and pass tests. Only Phase 3 (live capital, attribution & ops, through the independent G3 go/no-go) remains, plus a short list of PreP3 hardening follow-ups (§8).

The one thing to understand. The pipeline is now genuinely runnable: qe-cli exposes train (real MAP-Elites search → ensemble → G1 → sealed vintage), backtest (a sealed vintage over a window → deterministic result), and ingest; a second binary qe-server triggers/supervises those jobs and serves an authenticated SPA to trigger, monitor, and review runs in a browser. Everything is still paper/offline: there is no live order submission anywhere (the order path terminates in a kill-gated simulator; live HTTP/wss sit behind a default-off http feature).

2 · What it does today

Two executables ship today: qe-cli (the deterministic pipeline jobs) and qe-server (the admin backend that supervises those jobs and serves the SPA).

CommandWhat it actually does today
qe-cli version Prints quant-engine 0.1.0.
qe-cli train
  --config <file>		 Runs the real training pipeline: MAP-Elites/WFO search → discrete-DE ensemble → DSR/SPA validation → the G1 gate → seals a content-hashed vintage under the artifacts dir. Emits JSON-line progress (generation, MAP-Elites archive coverage, CV folds, best-so-far, G1 pass/fail). Deterministic for a fixed seed (same seed ⇒ same vintage id).
qe-cli backtest
  --vintage <id> --start --end
  --resolution … --run-dir …		 Loads + verifies a sealed vintage, builds decision bars via the shared feature pipeline, runs each chromosome through the backtester, weight-aggregates to ensemble returns, and writes a deterministic result.json (CAGR/Sharpe/Sortino/max-DD/win-rate/profit-factor, equity & drawdown curves, monthly heatmap, per-trade rows). JSON-line progress ends with done.
qe-cli ingest
  --config --start --end --resolution		 Populates the LMDB market store from an injectable historical source (real Binance decoders stay behind the default-off http feature). Also exposes the read-only coverage query the admin UI's Market-data view uses.
qe-server axum + tokio backend (a second composition root — reuses the training/shared crates, never qe-runtime). Google-OAuth login gated by QE_ADMIN_ALLOWED_EMAILS + signed cookie; a file-based run store (data/runs/<id>/…) that spawns & supervises qe-cli backtest/train subprocesses; /api for runs, vintages, and market-data coverage; and serves the built React SPA at / (same origin).

The React SPA (web/, Vite + TypeScript, the ported Claude Design system) gives an authenticated browser UI: Login, Backtests (list / new / result with live progress + the full result contract), a Training monitor (generations, archive-coverage grid, CV folds, best-so-far → G1 verdict → link to the sealed vintage's backtest), and read-only Market-data coverage. Trade/Risk surfaces are present but disabled (Phase 3).

The full runtime loop exists as tested libraries (Phase 2, through G2)

Beyond the runnable jobs above, the entire live/paper runtime is built and tested but is exercised through cargo test, not yet behind an operator command:

  • Venue-aware REST client + wss Market/Realtime/user-data streams (behind the default-off http feature)
  • Live kline source (REST-prime + wss-stitch), factor join, mark-EMA loop; the shared evaluator session (replay & live, no state copy), deterministic cold-start bootstrap + in-process cutover with restart parity
  • Circuit breakers, position netting, the Hedge Planner, pre-trade risk, the out-of-band kill switch, the venue adapter + position keeper + paper simulator, planner↔edge transport, vintage load + rollover
  • Real-time reconciliation divergence alarm and the G2 shadow/dry-run gate: would-be orders reconciled vs the simulator, nothing submitted

Test baseline: 668 Rust tests (+2 ignored) across the workspace, plus 40 frontend tests (Vitest/RTL), all green; firewall + decoupling guards green. There is still no live order submission anywhere — the order path terminates in a kill-gated paper simulator.

3 · How to run it

Prerequisites

The Rust toolchain pinned by rust-toolchain.toml (cargo on PATH) for the CLI/server; Node (for building the SPA). No database, no API key, no network needed to run the offline jobs against the committed sample store.

Run the pipeline (CLI)

# train the real pipeline → seals a content-hashed vintage (deterministic per seed)
cargo run -p qe-cli -- train --config config.example.toml
#   → JSON-line progress (generations → G1) → data/artifacts/vintages/<id>/

# backtest a sealed vintage over a window → deterministic result.json
cargo run -p qe-cli -- backtest --vintage <id> --start 2021-01-01 --end 2024-12-31 \
    --resolution 1h --run-dir /tmp/run --json

# populate the LMDB market store (real network decoders behind the default-off `http` feature)
cargo run -p qe-cli -- ingest --config config.example.toml --start … --end … --resolution 1h

Run the admin UI (server + SPA)

# 1) build the SPA
cd web && npm ci && npm run build      # → web/dist

# 2) run the server (serves web/dist at / and the API at /api)
QE_SERVER_STATIC_DIR=web/dist \
QE_ADMIN_ALLOWED_EMAILS=you@example.com \
QE_GOOGLE_CLIENT_ID=… QE_GOOGLE_CLIENT_SECRET=… QE_GOOGLE_REDIRECT_URI=… QE_SESSION_SECRET=… \
    cargo run -p qe-server
#   → sign in with Google (allowlisted) → trigger/monitor/review backtest & training runs

What you provide

Only a config file. config.example.toml works as-is. Every field is overridable by a QE_-prefixed environment variable (nested with __):

QE_DETERMINISM__SEED=7 cargo run -p qe-cli -- train --config config.example.toml

Config shape (TOML): profile, instruments / [[universe]] (with listed/delisted dates), [bars] (base + reconstructed resolutions), [storage] directories (relative, volume-friendly), [determinism].seed.

4 · Architecture

The workspace is 21 crates (+ a web/ SPA). The organising idea is a shared spine (qe-domain + qe-signal) that both the offline training world and the online runtime world depend on — and nothing else crosses between them. That separation is not a convention; it is checked by tests (see §6).

Crate map by layer

LayerCratesRole
Foundation
(cross-cutting)		 domain · error · config · determinism · clock · telemetry · risk Types (bars, instruments, time), error taxonomy, layered TOML config, the reproducibility/lineage harness, time-sync guard, structured logging, and the risk-limit / kill-switch contract.
Storage storage LMDB market-data & synthetic-feature stores (no database on the runtime hot path).
Shared spine domain · signal signal holds bar reconstruction, the indicator catalogue, feature assembly, regime labelling, and the strategy genome — the one piece of strategy logic both sides must run identically.

Admin backend
(2nd composition root)		 cli · server cli = the runnable train/backtest/ingest jobs (the composition root that may touch both worlds); server (qe-server) = axum+tokio, OAuth, run store + subprocess supervision, /api + static SPA. Async is isolated here; qe-server never depends on qe-runtime/qe-venue (checked by the firewall/decoupling tests).

Guards & root architecture · cli architecture hosts the information-firewall CI guard; cli is the composition root (the only crate that may touch both training and runtime).

How the dependencies actually wire (verified)

Crate dependency map. Training (left) feeds the shared spine; runtime (right) reads it and loads the sealed vintage read-only. qe-cli is the only crate that touches both sides.
  • qe-signal → qe-domain only. It is the narrowest crate and the spine of everything.
  • qe-runtime → {domain, risk, signal, storage, venue, vintage, determinism} — never wfo or ensemble.
  • qe-vintage → {signal, risk, determinism}. Because the strategy genome lives in signal (relocated in QE-207), a vintage can carry chromosomes to the runtime without dragging in the training crates — that is what keeps the two sides decoupled.
  • qe-cli depends on both worlds; it is the only place they meet, and it is the intended place.

5 · Data flow (end-to-end intent)

The spec numbers the pipeline ①→⑦ on the training side and ③/④ on the runtime side. The vintage is the artefact that bridges them.

The vintage artefact (⑦) bridges training and runtime: it passes the G1 gate, then the runtime bootstraps from it and runs the live loop toward the G2 dry-run gate. No orders are sent before G2.

6 · The two firewalls (why this codebase is shaped the way it is)

Two CI-enforced invariants drive nearly every architectural decision. A maintainer must understand both before moving code between crates.

Both sides may depend into the shared spine (green); neither may depend on the other directly (red). That is why the genome lives in qe-signal and the vintage is plain data.

① Information firewall QE-132

Test: crates/architecture/tests/firewall.rs — directional.

The evolutionary search must not reach into portfolio or live concerns. It asserts what qe-wfo and qe-ensemble are allowed to depend on (not the reverse): each is forbidden from reaching runtime, venue, or each other. This stops the search from ever "seeing" outcomes it must be blind to.

② Train/live decoupling QE-001

Test: crates/cli/tests/dependency_topology.rs — bidirectional.

qe-runtime must not transitively reach qe-wfo/qe-ensemble, and they must not reach qe-runtime. "The only code shared between the training and runtime sides crosses through signal/domain." This is why the genome lives in signal and the vintage is plain data.

Practical rule for contributors. If you need to share something between the training and runtime sides, it goes in qe-signal or qe-domain — nowhere else. If you find yourself wanting qe-runtime to depend on qe-wfo, stop: the decoupling test will fail, and the right move is to push the shared type down into the spine (exactly what QE-207 did with the genome).

8 · What's left to do

Phases 2 and PreP3 are complete. The full paper/sim runtime loop (through the QE-222 G2 shadow gate) and the admin platform (QE-251–261: runnable backtest/train/ingest jobs, the qe-server backend with OAuth + run supervision, and the React SPA) have all landed and passed review. Near-term — PreP3 hardening follow-ups QE-262–266

Surfaced by the PreP3 code reviews (recorded in docs/mds/reviewed/). None blocks anything — PreP3 shipped green. Priority: P1 = correctness (do before trusting training output); P2 = before wider exposure / heavier load; P3 = opportunistic.

TicketWhat & whyPriority
QE-262Persist catalogue version/states in the vintage; assert on load. Today train and backtest both use the single default catalogue, and Genome::is_valid only bounds-checks indices — a reordered/version-bumped catalogue would score genomes against the wrong schema silently. Seal a catalogue fingerprint and fail loudly on drift. P1 · correctness
QE-263Run-store startup reconciler. A server restart leaves in-flight runs wedged at running/queued forever; reconcile orphans to a terminal state on boot. P2

For someone evaluating this to invest

This system cannot trade with real capital today. Phase 2 is complete through the G2 shadow/dry-run gate — the full runtime loop runs against live data and produces would-be orders reconciled against a simulator — but the order path still terminates in a paper simulator: there is no live broker submission. The training/backtest pipeline is now runnable end-to-end (CLI jobs + an authenticated admin UI), but Phase 3 must still complete through the independent G3 go/no-go with a staged, capped capital ramp before any real capital is involved.

What is strong and verifiable: a disciplined, deterministic, information-firewalled design with deep test coverage, a friction-true backtest (fees + real funding + slippage so no strategy is ranked on returns that won't exist live), and statistical over-fit gates (deflated Sharpe, SPA) baked into the validation milestone. The engineering rigour is real; the product is mid-build.
