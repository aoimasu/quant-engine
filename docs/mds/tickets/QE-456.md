# QE-456 — `GET /api/vintages/{id}` vintage-detail read endpoint  *(R1 — inspection)*

> **Status:** Todo — [design QE-455](../../architecture/qe-455-research-flow-design.md#7-vintage-inspector--apivintagesid-surface) · [backlog index](../../backlog.md#research-flow) · [spec](../../specs.md)

`Phase: Research-flow (R1)` · `Area: backend / api` · `Depends on: QE-467, QE-257, QE-260`

**Why.** The ensemble composition, per-chromosome weights, and — after **QE-467** persists it — the full G1
gate/deflation evidence, holdout series handle, and data-provenance **now all live inside the sealed vintage**
(`VintageContent.chromosomes`/`weights`/`lineage` + QE-467's persisted evidence block, `crates/vintage/src/lib.rs:45`).
But `GET /api/vintages` (`crates/server/src/read.rs:68`) returns only a thin list (`VintageListItem{id,label,summary}`)
and there is **no detail endpoint**. `evolve` got a rich formula-pool inspection (`GET /api/formula-pools/{id}`,
PoolReview); the `train`/vintage path never did. This is the read half of closing that coverage gap. Additive —
**no engine change** and **no recompute** (it reslices exactly what QE-467 sealed).

**Scope / requirements.**
- Add `GET /api/vintages/{id}` beside the existing list, loading via `VintageRepository::load` (hash-verified,
  `crates/vintage/src/lib.rs:256`) on a `spawn_blocking` worker (as the list already does), session-authed.
- **Response body is resliced to EXACTLY what QE-467 persists — promise no data not sealed:**
  - **Ensemble composition** — for each `chromosomes[i]`: the referenced indicator ids
    (`Genome::referenced_features` resolved through the sealed `CatalogueIdentity`, distinguishing catalogue vs
    evolved formulas) and the aligned **weight** `weights[i]`.
  - **G1 gate / deflation evidence** — the persisted evidence block (DSR, uncensored PBO + population size,
    SPA p, IC/FDR, distinct-trial basis, cost-stress `min{1×,2×}` net, realised turnover, `capacity_usd`) **as
    sealed by QE-467**; the endpoint reads it, never re-runs `evaluate_g1`.
  - **`data_provenance`** (`real | synthetic | mixed`) from `VintageContent.lineage` (QE-467) — so the inspector
    can render the provenance banner.
  - **A ref/handle to the persisted net-of-cost holdout return series** (QE-467) — the endpoint returns the
    content handle, not the recomputed series.
  - **The frozen-holdout split + regime composition** — `{holdout_range, embargo, train_range}` and the holdout
    regime labels read from `VintageContent.lineage` (QE-467, design §4).
  - The provenance sidecars already in the content: `slippage`, `sizer`, `worst_case_loss`, `calibration`,
    `catalogue` identity.
- **Vintage→run reverse-join.** The vintage id is a **content hash**, so surfacing the producing run (steer
  delta, run status) requires scanning the run store for the run whose `train.vintage == {id}`. Specify a
  **deterministic tie-break** (e.g. earliest `created_at`, then lexicographic run id) since more than one run
  can seal the same content-identical vintage; the join is read-only and adds no new cross-crate edge.
- `404` for an unknown id; the same error-body shape the read module already uses.

**Out of scope.** The SPA screen (QE-457). Any recomputation of gate stats or the holdout series (read-only over
what QE-467 sealed — never re-runs the gate). Any promote/seal/select affordance (a vintage is already sealed by
the train gate; this endpoint is inspection only, per the design's leaderboard-is-not-selection boundary).
Ingest/flow work. Persisting the evidence itself (that is QE-467).

**Acceptance criteria.**
- `GET /api/vintages/{id}` returns composition (chromosomes→indicators + weights), the QE-467-persisted
  gate/deflation evidence, `data_provenance`, the holdout-series handle, and the holdout split + regime
  composition for a sealed vintage; the artefact is hash-verified on load. It promises no field QE-467 does not
  persist.
- The vintage→run reverse-join resolves the producing run by `train.vintage == {id}` with a deterministic
  tie-break; the endpoint recomputes nothing.
- No new `qe-server → qe-wfo`/`qe-ensemble` code edge is introduced (the server already depends on
  `qe-vintage`; this only *reads* the sealed artefact); the `firewall`/`dependency_topology` test stays green.
- An unknown id yields `404`; a corrupt/failing-verify artefact yields a `500` with a message, never a panic.
- The endpoint performs no gate recomputation and exposes no seal/promote/select action.
