# QE-465 — SPA ingest-trigger screen + provenance column in MarketData  *(R3 — real ingest)*

> **Status:** Todo — [design QE-455](../../architecture/qe-455-research-flow-design.md#83-the-spa-qe-465) · [backlog index](../../backlog.md#research-flow) · [spec](../../specs.md)

`Phase: Research-flow (R3)` · `Area: frontend` · `Depends on: QE-464`

**Why.** QE-464 exposes the ingest trigger + provenance; the operator needs to drive it and to *see* the
real-vs-synthetic origin at a glance. The MarketData view (`web/src/app/MarketData.tsx`) currently shows
coverage with no source marker.

**Scope / requirements.**
- An **ingest-trigger screen**: select instruments, date range, resolution, and a **fetch-all** toggle →
  `POST /api/ingest` (QE-464); after launch, surface the run's progress via the standard run monitor.
- **Progress + cancel for long real ingest.** A real historical ingest can run long — show **per-page /
  percentage progress** and expose a **cancel** affordance (reusing the run-halt/abort path). *(Note the
  `HistoricalSource::fetch() → one window` seam may need a streaming/paged change to emit incremental progress;
  flag it as a dependency risk if it lands as a follow-up.)*
- A **provenance column** in the MarketData coverage table showing `real` vs `synthetic` per coverage row
  (from QE-464's tagged `CoverageRow`). **Mixed-provenance rendering:** an instrument with interleaved
  provenance renders as **one row per provenance run** OR a **`mixed` badge with drill-down** to the
  per-provenance rows — never a single unmarked range. A **component test covers the interleaved case**.

**Out of scope.** The endpoint / run-kind / store provenance (QE-464). The real decoder (QE-463). Any editing
or deletion of stored data from the UI.

**Acceptance criteria.**
- The ingest screen submits a valid `POST /api/ingest` (instruments / date range / resolution / fetch-all) and
  shows per-page/percentage progress with a working cancel affordance.
- The MarketData table renders a provenance column marking each coverage row `real` or `synthetic`; mixed
  provenance renders as row-per-provenance-run or a `mixed` badge with drill-down, never one unmarked range.
- Component tests cover the trigger form, the progress/cancel UI, and the provenance column rendering including
  the interleaved-provenance case.

`Spec ref: QE-455 §8.3 "The SPA"; §8.2 (provenance visible so nobody trains on synthetic as real).`
