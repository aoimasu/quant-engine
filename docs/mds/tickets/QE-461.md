# QE-461 — Flow supervision: concurrency lane + resume/checkpoint/halt  *(R2 — composite flow)*

> **Status:** Todo — [design QE-455](../../architecture/qe-455-research-flow-design.md#53-resume--halt-the-long-composite-problem) · [backlog index](../../backlog.md#research-flow) · [spec](../../specs.md)

`Phase: Research-flow (R2)` · `Area: backend / orchestration` · `Depends on: QE-460, QE-407`

**Why.** A run today is a terminal 4-state `RunStatus` (`queued → running → succeeded | failed`,
`crates/server/src/runs/model.rs:16`) with **no resume** — fine for a minutes-long backtest, wrong for a
multi-hour composite flow. Building on QE-407's supervised-task registry / graceful-shutdown work, give the
long composite a dedicated concurrency lane and the ability to resume from its natural checkpoint and be halted.

**Scope / requirements.**
- **Dedicated concurrency lane** — a `QE_SERVER_MAX_FLOW_CONCURRENCY` semaphore (default 1), mirroring the
  `evolve` semaphore (QE-454 §13.10), so a multi-hour flow never starves interactive backtests; plus a per-flow
  wall-clock deadline reusing the QE-425/QE-407 `abort → kill_on_drop → terminally-mark` pattern.
- **Resume from the sealed-vintage checkpoint** — a flow that has sealed its vintage but not finished the
  backtest resumes **from the backtest phase** on server restart (via the QE-407 startup reconciler + task
  registry), rather than re-running the expensive search. The sealed vintage is the checkpoint (content-addressed,
  already durable); resume rides the **same** recorded seed + holdout split so it stays deterministic.
- **The resume predicate (specify it).** The reconciler distinguishes a **resumable** flow — *vintage sealed
  ∧ backtest incomplete*, discoverable from the `meta.json` sub-run ids + the recorded `train.vintage` — from a
  **dead** run (no sealed vintage, or backtest already terminal). It **re-spawns only the backtest phase** for a
  resumable flow; a dead run is terminally marked, never silently re-searched.
- **Authorised halt — reuse `Failed` + a halt reason, NOT a new status.** A `POST /api/runs/{id}/halt` arm for
  flows (mirroring the `evolve` halt, QE-454 §13.11) → SIGTERM → the run terminates as **`Failed` carrying a halt
  reason** — **no 5th `RunStatus` variant is added** (`halted` is a reason, not a status; matches the
  evolve/QE-452 halt shape). A partially-sealed vintage (if any) is retained and auditable.
- **No new run status leaks to the gate/vintage path** — resume/halt are supervision concerns riding the
  existing terminal 4-state `RunStatus`; the seal predicate and `evaluate_g1` remain the sole authority for what
  a vintage is.

**Out of scope.** The composite run-kind + holdout carve (QE-460). Intra-search (finer-than-sub-run)
checkpoints (design §14 open question — out of scope). The SPA Flow page (QE-462). Any change to the gate/seal.

**Acceptance criteria.**
- Flows run under the dedicated `QE_SERVER_MAX_FLOW_CONCURRENCY` lane; an over-limit flow queues rather than
  starving interactive backtests; a flow exceeding its deadline is terminally marked via the existing pattern.
- The resume predicate holds: a server restart with *vintage sealed ∧ backtest incomplete* (discoverable from
  `meta.json` sub-run ids + `train.vintage`) re-spawns **only** the backtest phase (no re-search), riding the
  recorded seed + holdout deterministically; a dead run is terminally marked, never re-searched.
- `POST /api/runs/{id}/halt` on a flow yields terminal **`Failed` with a halt reason** (no new `RunStatus`
  variant) and retains any partial vintage; a resumed or halted flow introduces no new run status into the
  gate/vintage path.
