# QE-467 — Persist seal evidence + net-of-cost holdout series + capacity/turnover + data-provenance into the sealed artefact (single `VINTAGE_FORMAT_VERSION` 7→8)  *(R1 — persistence foundation)*

> **Status:** Todo — [design QE-455](../../architecture/qe-455-research-flow-design.md#41-the-persisted-seal-evidence-qe-467--the-foundation) · [backlog index](../../backlog.md#research-flow) · [spec](../../specs.md)

`Phase: Research-flow (R1)` · `Area: vintage / server / storage / run-protocol` · `Depends on: QE-260, QE-434, QE-431, QE-440, QE-430, QE-437, QE-439`

**Why.** The downstream research-flow tickets (inspector QE-456/457, leaderboard QE-466, flow QE-460) were
specified to *read* provenance and gate evidence out of the sealed vintage — but that evidence is **computed at
gate time and thrown away, never carried into `VintageContent`**. `GateSnapshot`
(`crates/server/src/runs/model.rs:150`) lives on the run's `meta.json`, not the sealed artefact; the tradability
results (`min{1×,2×}` cost-stress, realised turnover, `capacity_usd`) computed by the capacity/ADV work
(QE-431/440) are not persisted; there is **no** per-vintage net-of-cost holdout return series for the leaderboard's
cross-vintage correlation to consume; and `VintageContent.lineage` (a bare
`Lineage{config_hash,input_snapshot_id,code_commit,seeds}`, `crates/determinism/src/lineage.rs:26`) has **no
data-provenance field**, so a synthetic-derived vintage is indistinguishable from a real one in the sealed
artefact. Every downstream ticket that "reads" this data would otherwise have to **recompute** it — re-opening
the deflation basis the QE-430..454 backbone sealed shut. This ticket persists it **once**, in **one**
schema/version bump, so the downstream tickets read (never recompute) the sealed evidence. **This is the R1
foundation — it lands first.**

**Scope / requirements.**
- **Persist the full seal evidence into the sealed artefact.** Extend the persisted gate evidence
  (`GateSnapshot` in `crates/server/src/runs/model.rs:150` and/or a new evidence block on `VintageContent`,
  `crates/vintage/src/lib.rs:45`) so the sealed vintage carries, in addition to the DSR/SPA/PBO already on
  `GateSnapshot`: the **IC/FDR** result (QE-434), the **cost-stress `min{1×,2×}` net figure** (QE-431/450 §4.6),
  the **realised turnover**, and the **`capacity_usd`** — i.e. the tradability results currently computed at
  gate time (QE-431/440) and discarded. The evidence is content-addressed (part of the hashed content).
- **Persist a canonical net-of-cost holdout return series.** Compute and persist, per sealed vintage, a
  **canonical net-of-cost holdout return series on the DEPLOYED capacity-capped weights** (QE-438 deployed-weight
  scoring, not gross / equal-weight / lone-Sharpe), content-addressed — the exact series the leaderboard's
  cross-vintage correlation (QE-430 R(N)/Fisher-z) and the inspector need. Store it by content handle so the
  detail endpoint (QE-456) returns a **ref/handle**, not a re-run.
- **Add hashed data-provenance + the extended lineage the flow needs.** Add a hashed
  **`data_provenance: real | synthetic | mixed`** field to `VintageContent.lineage` (extending `Lineage` or a
  sibling lineage block on `VintageContent`), plus the extended lineage fields the research flow requires:
  - the **holdout split** `{holdout_range, embargo, train_range}` (§4),
  - the holdout **regime composition** (the regime labels the holdout spans, QE-125),
  - a **per-holdout consultation count** (the overlap-keyed budget QE-460 increments),
  - the **steer delta** — indicator-subset hash + gens/pop + window/fold counts (what QE-458 recorded).
  Define the **whole** extended schema here. Downstream tickets **populate** these fields (QE-458 writes the
  steer delta, QE-460 writes the split/regime/consultation count); QE-467 **owns** the schema and the single bump.
- **Exactly one version bump.** Make this the **only** `VINTAGE_FORMAT_VERSION` 7→8 bump
  (`crates/vintage/src/lib.rs:41`): regenerate the vintage goldens via the **real seal path** (not by editing
  fixtures), and update the `assert_eq!(VINTAGE_FORMAT_VERSION, 7, …)` at `crates/vintage/src/lib.rs:533` to `8`.
  The QE-006 determinism harness must stay green. No downstream ticket bumps the version again — QE-460's lineage
  writes and QE-458's steer-delta writes ride **this** bump.

**Out of scope.** The read endpoint (QE-456) / inspector (QE-457) / leaderboard (QE-466) that consume the
persisted evidence. The flow that populates the split/regime/consultation fields (QE-460). The steer delta the
search records (QE-458). Any change to `evaluate_g1`, the seal predicate, or the gate thresholds (this persists
the gate's own outputs — it does not recompute or re-decide them). Any `PROTOCOL_VERSION` bump (that is QE-460's
run-kind concern, not the vintage schema).

**Acceptance criteria.**
- The sealed `VintageContent` carries the full seal evidence (IC/FDR, cost-stress `min{1×,2×}` net, realised
  turnover, `capacity_usd`) alongside the existing DSR/SPA/uncensored-PBO, all content-addressed.
- A canonical **net-of-cost holdout return series on the deployed capacity-capped weights** is persisted per
  vintage and addressable by handle; it is never gross/equal-weight/lone-Sharpe.
- `VintageContent.lineage` carries a hashed `data_provenance ∈ {real, synthetic, mixed}` plus the holdout split,
  holdout regime composition, per-holdout consultation count, and steer-delta fields (schema defined here;
  populated downstream).
- **Single bump / golden safety:** `VINTAGE_FORMAT_VERSION` is `8`; goldens are regenerated via the real seal
  path (the bump moves them **intentionally**); the `== 7` assertion is updated to `== 8`; the QE-006
  determinism harness is green and shows **no unrelated hash drift** (only the intended fields changed the id).
- No `qe-vintage → qe-wfo`/`qe-ensemble` code edge is added; the firewall/dependency-topology test stays green.
