### Design notes for the reviewer
- **One fold, shared by batch + streaming.** `BarReconstructor` is the single incremental fold;
  `reconstruct_batch` = push-all + finish. So batch and streaming cannot diverge — the QE-206 parity
  guarantee is structural, not a coincidence of two implementations.
- **Storage-free hot-path logic.** `qe-signal` stays `qe-domain`-only (no LMDB) — reconstruction runs
  identically in batch and in the latency-sensitive runtime (QE-003). The synthetic-store caching is
  a separate `qe-ingest` bridge.
- **Deterministic boundaries.** Window start = `floor_div(open_time, target_ms)·target_ms` — depends
  only on the timestamp + target resolution, never on batch size / arrival order / thread count.
- **Topology.** New `qe-ingest → qe-signal` edge is acyclic (signal is `qe-domain`-only); QE-001
  guard re-runs green.
- **Out of scope:** live-runtime streaming wiring (QE-205). The streaming `BarReconstructor` is
  exposed now so QE-206 can prove parity against the live path.

### Review notes

**Verdict: [Approved].** Reviewed strictly as architect + senior engineer against the full diff vs `main`
(head `1ca1897`) — read `reconstruct.rs`, the `recon.rs` bridge, both test files, and the Cargo/topology
edges. The AC is met in its **strongest** form and the design is clean.

**AC — batch == streaming parity (PASS, structural).** There is **one** incremental fold: `reconstruct_
batch` *is* `BarReconstructor::new` + push-every-bar + `finish`. The parity test runs that exact loop
manually and compares — so batch and streaming cannot diverge by construction (not two implementations
that happen to agree). I traced the 70-min fixture → 3 windows `[0,30)/[30,60)/[60,90)` incl. the partial
last window, all matching.

**Determinism (PASS).** `window_start = open_time.div_euclid(target_ms)·target_ms` is epoch-aligned
(correct for negatives via `div_euclid`) and depends only on the timestamp + target — independent of batch
size, arrival order, and thread count (`windows_align_to_epoch_boundary_not_first_bar` confirms a first
bar at +10m still opens the window at 0). The OHLCV roll-up is standard (open=first, high=max, low=min,
close=last, volume/trades summed); I hand-checked it (open 100, high 113, low 95, close 97, vol 70, trades
14) and the tier counts (48×5m → 8×M30 + 1×H4). Volume accumulates as exact `Decimal` and re-wraps via
`Qty::new` (non-negative sum, never fails); `Bar::new` re-validates the roll-up.

**`TargetNotMultiple` guard — acceptable / endorsed.** I independently confirmed every coarser tier in the
current enum (5/15/30/60/240/720/1440 min) is an integer multiple of every finer one, so the guard is
currently unreachable — but it is a **cheap, documented invariant** protecting boundary-alignment against a
future non-aligned resolution (e.g. a 45m or 10m tier). That is correct defensive engineering, not dead
code; keeping it (with the test's note) is the right call.

**Storage-free hot path + topology (PASS).** `qe-signal` stays `qe-domain`-only (deps: qe-domain /
rust_decimal / thiserror — **no LMDB**), so the reconstruction fold runs identically on the latency-
sensitive runtime path (QE-003) and in batch. The synthetic-store caching is a correctly-separated thin
`qe-ingest` bridge (`cache_reconstructed_tiers` → `reconstruct_tiers` + lineage-tagged `put_recon_bars`);
the integration tests prove the round-trip and that a different lineage reads stale. The new `qe-ingest →
qe-signal` edge is **acyclic** (signal is domain-only) and `qe-signal` is exactly the training/runtime
shared seam the QE-001 guard permits — the `runtime ⊥ wfo/ensemble` invariant is untouched.

**Verification caveat (transparency).** The Rust toolchain is absent from this review environment, so I did
not execute the gates (incl. `dependency_topology` and `cargo deny`). The verdict rests on full static
review + hand-traced tests and manifest-level confirmation of acyclicity. I did not rely on the PR's "all
green" claim; treat the gate results as developer-reported. Nothing in the review contradicts them.

**Trivial note (non-blocking, no action required):** `reconstruct_tiers` makes one full pass over
`base_bars` per tier (`O(tiers × bars)`); a single-pass fan-out across N `BarReconstructor`s would be
`O(bars)` with N folds. Irrelevant for the offline batch path — noted only for completeness.
