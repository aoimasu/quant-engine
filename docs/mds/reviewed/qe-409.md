## Review verdict — [Approved]
QE-409 completes the auth story across both surfaces with the four AC met and the security-sensitive paths sound under adversarial reading.

1. **Fail-closed secret (core).** `check_session_secret_policy(addr, ephemeral)` refuses iff `ephemeral && !addr.ip().is_loopback()`. Wired in `main.rs` on `cfg.addr` (the exact bind address) BEFORE `TcpListener::bind` and returns `ExitCode::FAILURE`. No bypass: `0.0.0.0` and dual-stack `[::]` both report `is_loopback()==false` ⇒ refused (safe direction); only `127.0.0.0/8`+`::1` are loopback; an IPv4-mapped `[::ffff:127.0.0.1]` is not loopback per std ⇒ also refused (still safe). Explicit `QE_SESSION_SECRET` passes on any bind; loopback keeps the AR-9 ephemeral fallback. `session_secret_policy_is_fail_closed_off_loopback` covers loopback v4/v6 ok, `0.0.0.0`/`10.0.0.5` refused, explicit-secret ok on all — non-vacuous.
2. **Cookie `Secure` conditional.** `cookie_secure_for` = `redirect_uri` scheme is https (case-insensitive, anchored on `://`). Derived once into `cookie_secure`; ALL four cookies (session, oauth-state, clear-state, logout) route through the single `set_cookie()` builder, which unconditionally emits `HttpOnly; SameSite=Lax; Path=/` and conditionally `Secure`. Can't silently drop `Secure` on a real https deploy — the only way is setting `redirect_uri` to http while serving https, which itself breaks the OAuth round-trip. Deriving from config (not the request scheme / `X-Forwarded-Proto`) is the robust, unspoofable choice. https/loopback tests non-vacuous.
3. **Logout.** `GET|POST /api/auth/logout` emits `Set-Cookie: qe_session=; …; Max-Age=0` with the same name/Path/attrs (overwrites), so subsequent `/api/me` is 401. Public + idempotent. `logout_clears_the_session_and_me_is_then_401` is genuinely login→200→logout→(emptied cookie)→401; `logout_is_reachable_without_a_session` covers the no-session path.
4. **401 → re-auth (FE).** Single choke point: `throwForResponse` emits `emitUnauthorized()` AND throws `UnauthorizedError` on every 401 across reads (`getJson`) and creates (`postRun`). Both pollers put `if (e instanceof UnauthorizedError) return;` BEFORE `failures += 1`, so a 401 stops immediately, burns no retry budget, sets no fatal error, no reschedule. `App.tsx` subscribes once (`useEffect([])`) and on signal does `setMe(null)+setStatus('unauth')` — no `window.location`/reload; Login remounts in the live tree. The initial `/api/me` probe uses `fetchMe()` (not the choke point) so first-load 401 doesn't double-fire. Vitest non-vacuous (exactly 1 fetch, `retrying='no'`, empty error, signal fired; App.test proves the authed shell flips in-place).
5. **QE-410 interaction.** Classification order auth → transient-retry → (success-path) terminal-status. A genuine transient 5xx is a plain `ApiError` (not `UnauthorizedError`), so it still increments the streak and retries to the cap; only 401 short-circuits. Both pollers' original QE-410 tests still pass.
6. **Scope.** QE-265 (OIDC nonce/JWKS/RS256/silent refresh) correctly untouched. `UnauthorizedError extends ApiError` (status pinned 401), so existing `instanceof ApiError` handlers keep catching; because `emitUnauthorized()` fires synchronously before the throw, the shell's unauth flip and the screen's catch batch in the same tick and the create screen unmounts — no stuck inline-error state.
7. **Panic-free / firewall / deps / golden.** No unwrap/expect on the auth or boot path; firewall green; zero Cargo delta; no golden touched.

### Non-blocking
1. **Logout is CSRF-reachable** (state-changing GET/POST, no CSRF token) — a cross-site forced request can log a user out. Severity low (forced-logout annoyance/DoS; no data exposure, nothing granted); dedicated CSRF-token infra is out of QE-409 scope (the OAuth login leg already has double-submit state). → tracked auth-hardening follow-up if logout ever grows side effects.
2. ~~`cookie_secure` assumes `redirect_uri` scheme == the true external scheme.~~ **Resolved (`e24d882`)** — `should_warn_insecure_cookies(addr, cookie_secure) = !addr.ip().is_loopback() && !cookie_secure` emits an advisory `tracing::warn!` in `main.rs` (after `check_session_secret_policy`, before bind). Correct predicate — fires only off-loopback + non-Secure, silent on loopback dev and any Secure/https config. Advisory only (a `warn!` in an `if`, no early return): never refuses boot, doesn't weaken the fail-closed check or cookie-Secure logic. `warns_only_for_non_loopback_insecure_cookies` non-vacuous across both axes.
