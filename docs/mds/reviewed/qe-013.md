### Design notes for the reviewer
- **Runnable skeleton, real vintage.** The training *stages* are P1 (QE-101+); QE-013 wires the
  composition root `run_train` that loads config (QE-002) → resolves the point-in-time universe
  (QE-012) → ensures the configurable state dirs exist → writes a **content-addressed vintage
  manifest** built from a real `qe_determinism::Lineage` (QE-006): `vintage_id = SHA-256(lineage)`
  validated via `qe_domain::VintageHash`. It produces a resolvable vintage from real inputs (config
  hash + seed + universe), which is what AC #1 asks of the P0 skeleton.
- **Deterministic / reproducible.** `code_commit` is passed in (binary supplies `QE_CODE_COMMIT` →
  crate version fallback); the manifest carries **no wall-clock**, so the same config + commit
  reproduces the same vintage id and bytes. Full multi-stage vintage lineage is QE-129.
- **12-factor packaging.** All state dirs come from `[storage]` (relative defaults), env-overridable
  via `QE_STORAGE__*`; nothing absolute is baked in. The `Dockerfile` (multi-stage `rust:1.96` →
  slim) runs the same `qe` binary with `/app/data` as a mounted volume — no platform lock-in.
- **Minimal deps:** hand-rolled arg parsing (two flags) instead of pulling `clap`, matching the
  crate's existing minimal-dependency ethos.
- **Testability:** logic lives in `src/lib.rs`; `main.rs` is a thin dispatcher — so `run_train` and
  the parser are unit/integration-tested.
- **Topology:** `qe-cli` is the composition root (already depends on the pipeline crates); adding
  `qe-config`/`qe-determinism` adds no edge into `runtime`'s forbidden set → QE-001 guard green.

### Review notes

**Verdict: [Approved].** Reviewed strictly as architect + senior engineer against the full diff vs `main`
(head `9098602`). All three ACs are met — by construction and by test — the vintage is genuinely
deterministic, and the composition-root topology is sound.

**AC #1 — one-command run → vintage (PASS).** `run_train` (crates/cli/src/lib.rs) loads config →
resolves the QE-012 universe (which validates listing windows) → creates the configured state dirs →
builds a real `qe_determinism::Lineage::from_config(cfg, "", code_commit, [seed])` → `vintage_id =
VintageHash::new(lineage.id())` (defensively `map_err`'d though SHA-256 is always 64-hex) → writes
`<artifacts_dir>/vintages/<id>/manifest.json`. **Determinism verified at the source level**, not taken
on faith: `Lineage` has exactly four fields — `config_hash`, `input_snapshot_id`, `code_commit`, `seeds`
— **no wall-clock**, and `id()` is SHA-256 over canonical (fixed-field-order) JSON; the manifest embeds
only those plus `schema`/`profile`/`universe` (all_known in config order), serialised via stable
`to_vec_pretty`. So same config+commit ⇒ identical id and byte-identical manifest; a different commit
flips the id. `content_hash` includes `profile` (forced by `Config::load`), so profiles don't collide.
Covered by `run_train_produces_a_vintage_manifest`, `vintage_is_deterministic_for_same_inputs`,
`example_config_loads_and_validates`, and the README documents the exact `cargo run -p qe-cli -- train
--config config.example.toml` step (AC #1's "documented steps").

**AC #2 — configurable, no absolutes (PASS).** Every state dir comes from `[storage]` (relative defaults
in `config.example.toml`: `data/lmdb/market` etc.), env-overridable via `QE_STORAGE__*`; `run_train`
hard-codes no path and creates only what config names. `all_state_is_under_configured_dirs_no_absolutes`
asserts the run writes only under the configured root and that all default storage paths are relative.

**AC #3 — Docker parity (PASS).** `Dockerfile` builds `cargo build --release --locked -p qe-cli` and runs
the **same** binary via `ENTRYPOINT ["qe"]`; `crates/cli/Cargo.toml` sets `[[bin]] name = "qe"`, so the
container entrypoint is byte-for-byte the local `qe`. State lives under the `/app/data` volume with
config-driven paths — no platform lock-in. Structural test `dockerfile_runs_the_same_binary` pins both
strings (Docker can't run in CI — correctly noted).

**CLI / structure (PASS).** Logic in `lib.rs`, `main.rs` a thin `ExitCode` dispatcher; hand-rolled
`parse_args` (no clap) with good coverage — defaults, flag parsing, and rejection of unknown
flag/command/profile and missing flag value. `code_commit()` reads `QE_CODE_COMMIT` with a
`CARGO_PKG_VERSION` fallback, so a vintage is always attributable and deterministic per build.

**Topology (PASS).** Read the guard (`crates/cli/tests/dependency_topology.rs`): it enforces
`runtime ↮ wfo/ensemble` via `cargo metadata` (normal edges only). `qe-cli` is the composition root and
already aggregates the pipeline crates; the added `qe-config`/`qe-determinism` reach only
`{qe-config, qe-domain}` (low-level leaves, no edge to runtime/wfo/ensemble), so no forbidden edge is
introduced — the guard stays green.

**Verification caveat (transparency).** I could **not** independently re-run the cargo gates this pass:
the Rust toolchain is absent from this review environment (no `cargo`/`rustc`/`rustup`). The verdict
rests on full static review of all changed source, source-level confirmation that `Lineage` carries no
wall-clock, and reading the topology guard. I did not rely on the PR's "all green"/end-to-end claim as
evidence; treat the reported gate + `cargo run` results as developer-reported. Nothing in the static
review contradicts them.

**Forward-looking notes (non-blocking — not AC defects):**
1. **`input_snapshot_id` is `""`** until the ingest stages land (QE-101+), so the vintage id does not yet
   bind input data — two runs over *different future data* with the same config+commit+seed would
   currently collide on id. Correct and explicitly deferred for the P0 skeleton; flagging so QE-101+
   threads the real snapshot id into the lineage before any data actually varies.
2. **`run_train` eagerly creates `market_dir`/`synthetic_dir`** which aren't consumed until P1. Harmless
   and consistent with AC #2 ("ensure the configurable state dirs exist") — noted only for awareness.
