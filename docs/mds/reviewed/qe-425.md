## Review verdict — [Approved]
1. **413 body cap (core) — empirically non-vacuous.** `DefaultBodyLimit::max(256 KiB)` is layered on the runs sub-router; the only body-carrying route is `POST /api/runs` (sibling GETs bodyless). `over_limit_body_on_post_runs_is_413` drives the PRODUCTION `build_router` with a 300 KiB body → 413, and `within_limit_body_on_post_runs_reaches_the_handler` proves a normal body passes and 400s on validation (not a false 413). Confirmed the 413 comes from THIS cap, not axum's 2 MiB default: deleting the `.layer(DefaultBodyLimit::max(…))` makes the 300 KiB body return 201 (a run is created), so the test fails without the cap. 256 KiB reasonable (~20–50× headroom for a few-KB run spec).
2. **408 timeout (core).** `TimeoutLayer::with_status_code(REQUEST_TIMEOUT, 30s)` on the `hardened` sub-router. `api_timeout_layer_returns_408_for_a_slow_handler` exercises the exact layer type at a 50 ms deadline: `/slow` (sleeps 30s) → 408, `/fast` → 200 control. A real 30s deadline is untestable in-process; layer mapping + structural placement + happy-path integration is an adequate Effort-S proof. 30s comfortably exceeds the slowest legitimate handler (coverage scan, `spawn_blocking` reads, OAuth token exchange).
3. **Health/static excluded (critical) — verified by router structure.** `GET /api/health` is `.route`d directly on `api` OUTSIDE the `hardened` group carrying the `TimeoutLayer` — never subject to the deadline. Static/SPA is the outer router's `fallback_service` (`Router::new().nest("/api", api).fallback_service(static_service)`), entirely outside `/api`, so it gets neither timeout, trace, nor body cap. `happy_path_is_intact_health_and_normal_get` confirms both `GET /api/health` and a normal authed `GET /api/runs` return 200.
4. **Concurrency deferral — acceptable.** Spec makes the global concurrency limit optional. The run-pool semaphore already bounds the genuinely expensive work (CLI-job spawning) + returns 503 on shutdown; a plain `ConcurrencyLimit` only queues, and `LoadShed + HandleError` is disproportionate for Effort-S. The deadline AC accepts 408 OR 503 and is satisfied by the 408.
5. **No regression.** `main.rs` untouched → QE-407 graceful shutdown independent/intact. `require_session` sits in `protected` inside `hardened`, so auth runs INSIDE the timeout. The `trace` layer is outermost on `api`, wrapping the inner `TimeoutLayer`, so a timed-out 408 still flows through TraceLayer and is logged (QE-413 preserved). QE-411 list path / QE-419 boot untouched. Firewall green.

### Non-blocking
1. The 408 is not proven end-to-end through the production `build_router` — no production handler hangs deterministically and none is injectable into the prod router, so the deadline is asserted via the exact layer type over a test route + structural placement. Reasonable Effort-S tradeoff.
2. The body cap is applied to the whole runs sub-router including its bodyless sibling GETs — harmless (a no-op for bodyless requests) and documented.
