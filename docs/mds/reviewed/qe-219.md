### Goal
Runtime loads the ensemble repo + calibration profile **read-only** at startup; periodic **rollover** replaces
the vintage in place when training emits a new one, without violating the firewall.

### Acceptance criteria (from backlog)
- [x] Startup loads a vintage read-only; a rollover swaps it atomically with lineage recorded —
  `startup_loads_vintage_read_only`, `rollover_swaps_in_place_with_lineage_recorded`,
  `rollover_rejects_unverified_vintage_keeping_current`.

### Implementation summary
- `crates/runtime/src/vintage_rollover.rs`: `ActiveVintage` holds the current sealed `Vintage` + a
  `RolloverRecord` history. `load(repo, id)` uses `VintageRepository::load` (open + hash-verify, never write) —
  read-only startup. `rollover(next)` / `rollover_from(repo, id)` **verify before** the single `current = next`
  swap (atomic: a bad vintage never becomes active; repo + calibration — calibration lives inside
  `current.content` — swap indivisibly). Every rollover records `from/to vintage_id` + `from/to Lineage`.
  `rollover` verifies unconditionally (single safety boundary) and imposes no same-id guard by design (rollback
  / re-emission are legitimate).
- Promotes `qe-determinism` (`Lineage`) to a direct dependency — cross-cutting (QE-006), not on either side of
  the QE-132 firewall, already transitive via `qe-vintage`.

### Review notes — [Approved] (two passes)

**Pass 1 (@ `98558aeb`) — [Approved].** AC genuinely met, no correctness/design defect. Read-only load is real
(`ActiveVintage` exposes no `repo.write` path; the read-only test snapshots on-disk bytes before/after —
non-vacuous). Rollover is atomic (verify-before-commit; the record's `from_*` is taken from the outgoing vintage
before the swap; the rejection test tampers `content.weights[0]` post-seal so `verify()` genuinely fails, then
asserts current unchanged + empty history). Lineage recorded on every transition. Scrutinise answers: (1)
verify-before-commit is the correct atomicity reading for a single-threaded runtime; Arc-swap correctly
deferred; (2) read-only proof sufficient; (3) unbounded history acceptable at rollover cadence; (4)
`qe-determinism` dev→direct promotion is the honest call (it defines `Lineage`) and adds no train→live firewall
edge; (5) accepting any verified vintage (no monotonic/same-id guard) is the right boundary — preserves
rollback. Two non-blocking observations: **O1** `rollover_from` verifies twice (defensible defence-in-depth);
**O2** a same-id rollover records a `from == to` transition (harmless).

**Pass 2 (@ `6705c90a`) — [Approved] confirmed.** O1/O2 addressed **doc-only** (logic byte-identical): O1 —
`rollover`'s doc states it verifies unconditionally as the single safety boundary, so `rollover_from`'s second
verify is deliberate defence-in-depth; O2 — `rollover`'s doc + design-note Risks explain a same-id vintage with
changed content is a real (differing-hash) transition, honestly recorded, so there is deliberately no same-id
