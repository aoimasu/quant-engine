## Review verdict — [Approved]
AC-1 (no blocking fs on an async handler body) is met. `list_runs`, `get_run`, `get_result`, and `create` no longer call `std::fs::` on an async body: `list_runs` runs its index read + per-run `read_meta` loop inside ONE `spawn_blocking(move || list_runs_blocking(&store))` (batched, not per-run); `get_run` wraps `store.read_meta`; `get_result` wraps the whole `read_meta` → `read_result` sequence via `read_result_outcome`; the result read moved into `RunStore::read_result` so the fs primitives all live in `store.rs`. The `handlers_do_no_blocking_std_fs` guard is non-vacuous and not trivially self-matching: it `include_str!`s `api.rs`, scans only the handler region (`split("#[cfg(test)]").next()`), strips `//` comment lines, and builds the `std::fs::` needle at runtime so the assertion isn't its own match.

AC-2 parity (critical) holds with no semantic drift. `list_runs_blocking` keeps the same `read_index`, `.iter().rev()` newest-first order, `Ok(None)` skip-on-missing-meta, error strings, and `Json(Vec<RunMeta>)` body — only relocated into one closure. `get_result` is byte-identical to `origin/main`: the new `ResultOutcome` mapping reproduces every case exactly — 200 `([(CONTENT_TYPE, "application/json")], bytes)`, 404 `not_found`, 409 not-ready `{"error":"result not available","status":...}`, 409 missing `{"error":"result artefact missing"}`, 500 `"failed to read run `{id}`: {e}"` — the only additions are the join-error → 500 fallbacks.

QE-407 interaction is sound. `create` offloads `init_run` to `spawn_blocking` before taking the lock (ordering preserved), then holds the async `index_lock` across a single `spawn_blocking(read_index → push → write_index).await` — the closure is `Send`, it is the sole awaited work under the guard, and the QE-407 registry lock is taken later, outside the `index_lock` scope (no nesting, no deadlock). Concurrent-create serialisation is unchanged; the executor thread is now freed instead of blocked. The registry/drain/shutdown tests still pass. Panic-free on the request path: every `spawn_blocking(...).await` has an explicit `Err(_) => internal(...)` arm, and `create` uses `.map_err(std::io::Error::other)??` — no `unwrap`/`expect`. Firewall/scope respected: no qe-* edge, blast radius confined to the run-store files (`api.rs`, `manager.rs`, `store.rs`, `tests/runs.rs`), no Cargo/golden/format change.

- nit (non-blocking): the `handlers_do_no_blocking_std_fs` guard is token-specific — it catches literal `std::fs::` but not an aliased `use std::fs::read; … read(…)` reintroduced in a handler. The real protection is that the fs primitives live in `store.rs` behind `spawn_blocking`; consider a clippy `disallowed_methods` rule on the handler module for a stronger, semantic guard.
- nit (non-blocking): out of QE-411's scope — the QE-407 shutdown-drain path (`terminally_mark_interrupted` → blocking `store.read_meta`/`write_meta`) still runs synchronous `std::fs` on the async executor during `shutdown`. Correctly left alone (its AC targets handler bodies, one-time process-exit path), but offload if shutdown latency under many in-flight runs ever matters. → QE-407 follow-up theme.
