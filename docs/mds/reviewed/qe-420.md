## Review verdict — [Approved]
Determinism is clean: `code_commit()` has a single caller (`crates/cli/src/main.rs:217`, live `run_train`), so no id-pinning test/golden touches the changed fallback. `train_job.rs` uses `Lineage::new(…, "qe-260-commit", …)`, `backtest_job.rs` uses `"fixture-commit"`, and `sample_vintage.json`/`golden_result.json` carry static `code_commit: "fixture-commit"` + pinned `content_hash`, read in place. Diff touches only `build.rs` (new), `main.rs`, `Dockerfile`, `tests/train.rs`, evidence — no fixture/golden regenerated. build.rs degrades gracefully (missing `git`/`.git` → `"unknown"` sentinel, never fails build); precedence and Dockerfile ARG→ENV correct. `--allow-dirty` seal-refusal was spec-optional and acceptably deferred (the `-dirty` SHA suffix already surfaces dirty builds).

Non-blocking nit (recorded, not actioned): build.rs watches `.git/HEAD` and `.git/packed-refs` but not the resolved loose branch ref (`.git/refs/heads/<branch>`), so a plain `git commit` on the current branch can leave a stale `QE_BUILD_GIT_SHA` on *incremental* builds until a clean rebuild. Low impact — the authoritative sealing path (release/Docker) uses the explicit `QE_CODE_COMMIT` override and CI/release does clean builds. Add a ref watch if local incremental drift matters.
