### Goal
The Edge gateway's execution mechanics consume Realtime-tier streams (`bookTicker`, `depth20@100ms`)
plus `aggTrade`. Subscribe them via the connection registry, disjoint from the Hedge-Planner
(Market-tier) data path by construction.

### Acceptance criteria (from backlog)
- [x] Edge-side and Planner-side streams share no upstream data path (verified) — test
  `edge_and_planner_streams_are_disjoint` (one registry, Market + Realtime subscribed; distinct tiers,
  one connection per tier, no subscription bleed either direction).

### Implementation summary
- `StreamChannel` + `BookTicker` / `Depth20` / `AggTrade`, venue-correct `stream_name()` forms.
- `Subscription::tier()` routes the three to `StreamTier::Realtime`; kline/markPrice stay `Market`.
- `cadence_ms() -> Option<i64>`: `Some` for fixed-cadence (kline/markPrice/depth20@100ms), `None` for
  event-driven (bookTicker/aggTrade). Registry does cadence gap detection only when a cadence is
  declared; event-driven outages still surface via `PumpOutcome.reconnected`.
- Convenience constructors `book_ticker` / `depth20` / `agg_trade`.

  `Kline`/`MarkPrice` still return `Some(...)`; only `BookTicker`/`AggTrade` (`None`) skip cadence gap
  detection — correct for event-driven streams, with outages still surfaced via `PumpOutcome.reconnected`.
- **Signature blast radius is contained.** `cadence_ms` has no callers outside the venue crate.
- **Venue-correct stream names** — `btcusdt@bookTicker`, `btcusdt@depth20@100ms`, `btcusdt@aggTrade`.
- **The AC is genuinely proven** by `edge_and_planner_streams_are_disjoint`: distinct tier routing,

- **Option-path test coverage** — `depth20_reconnect_reports_gap_but_event_driven_does_not` exercises both
  branches (depth20 reports a gap across reconnect; aggTrade flags `reconnected`, no gap).

Non-blocking observations (advisory only — none block merge; no action taken, recorded for forward awareness):
1. depth20's 100 ms cadence with the strict `>` gap threshold is more jitter-sensitive than the coarser
   Market streams. Faithful extension of the existing QE-202 model; `gap` is informational, not fatal.
   Follow-up only if a downstream consumer treats depth20 gaps as actionable (consider a slack term then).
2. `depth20@100ms` is hard-coded as the only depth flavour (already scoped out in the design note). If future
   tiers need other flavours, `Depth20` as a unit variant will need to carry level/interval.
