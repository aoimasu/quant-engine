## Review verdict — [Approved]
1. **Image runs server+SPA end-to-end (core).** Three stages correct: (1) `node:20-slim` builds SPA → `/web/dist` (deps before source for cache; `npm ci` + `npm run build` → vite `dist`); (2) `rust:1.96-bookworm` builds `qe-server --features http` AND `-p qe-cli`; (3) `debian:bookworm-slim` installs `ca-certificates` (native-tls/OIDC), copies `qe-server` + `qe` into `/usr/local/bin/`, SPA into `/app/web/dist`. Verified beyond the live run: qe-cli declares `[[bin]] name="qe"` (so `COPY …/release/qe` resolves), the two binaries are co-located in `/usr/local/bin/` — exactly what `ServerConfig.cli_bin = resolve_cli_bin()` expects for `run_manager`'s spawner; `QE_SERVER_STATIC_DIR=/app/web/dist` is the real env var; `EXPOSE 8080` matches `QE_SERVER_ADDR`. `--features http` is the right deploy build — the default wires `DisabledVerifier` (login can't complete); the live-OAuth verifier is behind `http`.
2. **Fail-closed (core).** `ENV QE_SERVER_ADDR=0.0.0.0:8080` is non-loopback → QE-409's `check_session_secret_policy` (unchanged, reused) refuses boot without `QE_SESSION_SECRET` (`session_secret_policy_is_fail_closed_off_loopback`); confirmed exit 1 live. Defense-in-depth: compose marks `QE_SESSION_SECRET: "${…:?…}"`, failing interpolation before the container starts. A misconfigured deploy cannot silently run on an ephemeral secret.
3. **Distinct image / no regression.** CLI `Dockerfile` (`ENTRYPOINT ["qe"]`, batch) untouched; separate `Dockerfile.server` (`ENTRYPOINT ["qe-server"]`). Zero Rust/crate/Cargo/golden files in the diff → gate + firewall unaffected by construction.
4. **Relative volumes + TLS scope + env docs.** `docker-compose.server.yml` mounts the RELATIVE `./data:/app/data` (QE-013, keeps QE-311 mechanical). TLS documented as an out-of-scope fronting-proxy assumption, tied to QE-409 (set `QE_OAUTH_REDIRECT_URI` to the public https callback so cookies mint `Secure`; README references the `should_warn_insecure_cookies` advisory). All auth/OAuth/`QE_SESSION_SECRET` env in `.env.server.example` (blank) + the README env table.
5. **`.dockerignore` / lean image.** Excludes `target/`, `web/node_modules|dist|coverage|.vite`, `.git/`, `data/`, `.env`/`.env.*` — with `!.env.server.example` re-included (the only shipped env file, no secrets). Real env files + run-store never enter the context; SPA rebuilt from source. 143 MB reasonable.
6. **QE-420 provenance under a `.git`-less context — correct, not a gap.** `QE_CODE_COMMIT` is declared in runtime stage 3 (not builder stage 2; `.git` dockerignored); `crates/cli/src/main.rs` reads it from the environment at RUNTIME (build.rs git-SHA is only a fallback). So the stage-3 `ENV QE_CODE_COMMIT` is inherited by both `qe-server` and the spawned `qe`, stamping the image's build commit into vintage lineage. Compose threads it via `--build-arg`. No provenance hole.
7. **CLI-image glibc — leaving it alone is correct scope.** The existing CLI `Dockerfile`'s unpinned `rust:1.96` vs `bookworm-slim` glibc mismatch is a latent pre-existing defect; the server image correctly pins `rust:1.96-bookworm`. Touching the CLI image would be scope creep — deferred (non-blocking 1).
