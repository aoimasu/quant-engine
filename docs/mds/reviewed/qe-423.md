## Verification

## Review verdict — [Approved]
Type-level deliverable verified empirically (not just "tsc passes"). `npx tsc -b` passes clean; adversarially bypassing the constraint (flipping the test's bad `key: 'nope'` to a real `key: 'id'` so the `@ts-expect-error` becomes unused) makes `tsc -b` emit `error TS2578: Unused '@ts-expect-error' directive` — proving the guard is live and gated by `npm run build` (`tsconfig.app.json` `include: ["src", …]` covers the keytyping test; `noUnusedLocals` on). So a real key typo fails CI, exactly as the AC requires.

1. **No casts (core).** All four call sites pass real types with the `as (X & Record<string, unknown>)[]` casts removed: `MarketData` (`Column<CoverageRow>`), `BacktestResult` (`Column<Trade>`), `BacktestsList` + `TrainingList` (`Column<RunListItem>`). Repo-wide grep finds zero `& Record<string, unknown>` in production `.ts`/`.tsx` (only test-local `Row` decls remain).
2. **Compile-error guard (core).** `Column<Row>` is a distributive union `{ [K in keyof Row & string]: KeyedColumn<Row,K> }[…] | DerivedColumn<Row>`, so `key` is a discriminant: a non-existent key matches no member and fails to compile. `render`'s `value` is contextually `Row[K]` — the test's `badValue` (`key:'id'` with `v.toFixed(2)`) is caught because `Row['id']` is `string`. Both `@ts-expect-error`s are real, exercised directives.
3. **Derived-column soundness (smoking gun for the ticket's value).** TrainingList's `gen`/`g1` were previously `key: 'gen'`/`key: 'g1'` — NOT real keys of `RunListItem` — which compiled silently (yielding `undefined`) only because their `render` ignores the value and reads `row.train?.generation`/`row.train?.gate`. The new typing rejects those fake keys, forcing the honest `id:'gen'`/`id:'g1'` `DerivedColumn` variant. Render bodies unchanged ⇒ output byte-identical. The union is watertight: `KeyedColumn` has `id?: never`, `DerivedColumn` has `key?: never` + required `render` — no member can be mixed.
4. **Internal cast confined.** The single `c.render as ((value: unknown, row, index) => ReactNode) | undefined` (plus the `value as ReactNode` fallback) lives entirely inside the component (a union-of-per-key-render-signatures can't be invoked with one runtime value). It does not leak to call sites — those stay fully checked (the `badValue` compile error is the proof). Minimal and necessary.
5. **No runtime change / a11y intact.** Keyed cell reads `row[c.key]`; derived cell value is `undefined` and renders from `row` — identical to prior behaviour. QE-422 a11y untouched (`role`/`tabIndex`/keydown, `DataTable.test.tsx` green). Dead sort CSS (`th.is-sortable`, `.qe-table__sort`) removed with zero remaining references; no sort feature added.
6. **Frontend-only / no golden.** Diff entirely under `web/` + `docs/`; no Rust, no golden/fixture.

### Non-blocking (minor, mostly pre-existing)
1. A `KeyedColumn` with no `render` still falls back to `value as ReactNode`, and `Row[K]` isn't constrained to be React-renderable (an object/array field would render poorly). Pre-existing (old code had the same `row[c.key] as ReactNode`), not worsened; a future tightening could constrain render-less keyed columns to `ReactNode`-compatible values.
2. `DerivedColumn.id` (used only for the React `<th>`/`<td>` key) isn't required to be unique across columns — two derived columns sharing an `id` would collide at the React-key level. Not type-enforceable and harmless today (ids are distinct).
