### Goal
Decisions flow planner→adapter over gRPC; fills/positions/heartbeat flow back. Backpressure and reconnection
handled; the QE-301 journal-append path must **never gate** the dispatch.

### Acceptance criteria (from backlog)
- [x] A target revision reaches the adapter and fills/positions return; the append path (QE-301) never gates
  this dispatch — `target_revision_reaches_adapter_and_fills_return`, `append_never_gates_dispatch`.

### Implementation summary
- `crates/runtime/src/transport.rs`: `PlannerAdapterLink<A: AppendSink>` — a **deterministic, single-threaded,
  pull-based** model of the planner↔adapter gRPC bidi stream. `TargetRevision` (absolute, monotonic `seq`) in;
  `AdapterReport::{Fill, Position, Heartbeat{ack_seq, health}}` (with `VenueHealth`) back.
- `pump()` → `plan_delta` vs the authoritative kept position → submit **through the QE-216 `VenueKillGate`** →
  absorb the fill into `VenueKeeper` → return fills + authoritative position + heartbeat. Heartbeat health is
  derived directly from the kill (`Down(kill_reason())` iff tripped), independent of delta activity.
- **Backpressure = coalesce-to-latest** (`dropped_superseded` observable) — lossless because targets are
  absolute. **Reconnection = re-snapshot (reconciled into the keeper) + re-send latest** — idempotent, no
  double-fill. **Append never gates dispatch** — the `AppendSink` is offered already-produced reports; its
  result only bumps `append_failures`. Real tonic/gRPC wire deferred to the runtime binary (QE-201/202
  offline-core convention); no new workspace dependency; firewall unaffected.

### Review notes — [Approved] (two passes)

**Pass 1 (@ `2e2986c8`) — [Reviewed].** AC #1 confirmed and the QE-301 append path confirmed *structurally*
non-gating (reports produced and returned before `append` is offered them). Backpressure coalesce-to-latest
sound (lossless for an absolute target). Raised **F1 (blocker):** heartbeat health misreported `Ok` while the
kill was tripped when the delta was zero — `health` was only set `Down` inside the `plan_delta = Some` branch
on a failed submit, so an at-target revision (steady state) while killed reported `Ok`, hiding the out-of-band
halt from the planner. Plus **F2 (nit):** `reconnect` snapshotted from the sim while idempotence rested on the
keeper — coincidental sync, not reconciled. **F3 (nit):** `keeper_mut()` leaked full `&mut VenueKeeper`,
enabling exactly the desync F2 warned about. Scrutinise answers noted that dropped/coalesced revisions never
reach the append sink (audit-trail caveat, QE-301 follow-up).

**Pass 2 (@ `3ff14ff7`) — [Approved].** All three resolved:
- **F1** — `apply_revision` derives `health` directly from `self.gate.kill().is_tripped()` (→
  `Down(kill_reason())` / `Ok`), outside the delta block; regression `at_target_revision_reports_down_while_killed`
  asserts `Down` for an at-target killed tick, and the normal-fill path still asserts `Ok`. `kill_reason()`
  reuses the QE-009 `OrderGate` fallback so the heartbeat reason matches `KillHalt.reason`.
- **F2** — `reconnect()` reconciles the sim snapshot into the keeper (`keeper.apply(&UserDataEvent::Position(
  report))`) before returning it, so snapshot and planning truth are one source (structural, not coincidental);
  test 4 confirms no double-fill on resume.
- **F3** — `keeper_mut()` removed; only narrow `observe_mark`/`observe_balance` remain, neither of which moves
  the kept position. With fills the sole position-mover, the sim-as-execution / keeper-as-truth invariant
  cannot be violated externally.
- No new issues; still single-threaded/deterministic; no new workspace dependency; firewall unaffected. AC #1
  holds. **Approved.**
