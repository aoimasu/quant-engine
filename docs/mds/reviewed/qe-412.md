## Review verdict — [Approved]
AC no-decode (core) is proven two ways, both sound. Type-level: `coverage_bounds` iterates `self.bars.remap_data_type::<DecodeIgnore>().prefix_iter(&rtxn, &prefix)`, whose items are `(key, ())` — `DecodeIgnore`'s decoder returns `()` without reading value bytes, so `SerdeJson<Bar>` is never invoked (compile-time guarantee; the closure destructures `(key, ())`). Behavioural: `coverage_path_never_decodes_bar_values` plants `b"\xffnot-json"` as a bar's value via a raw `Bytes` remap while keeping its valid `bar_key`, then asserts `coverage_bounds`/`coverage` return the correct `(100_000, 200_000, 2)` while `scan_bars(MIN..MAX)` errors on the very same store — the control makes it non-vacuous: `0xff`-prefixed non-JSON genuinely fails `SerdeJson<Bar>` decode, so coverage's success can only mean it never touched the value.

AC byte-identical output holds by key-layout reasoning and the unchanged fixture test. `bar_key = bar_prefix(instrument, resolution) ‖ order_i64(open_time.millis())`, so every key ends in the order-preserving big-endian encoding of `open_time`; keys sort chronologically within a prefix, and `time_from_key` (trailing 8 bytes via `unorder_i64`) recovers exactly the `open_time` the old path read from `bars.first()/last().open_time()`. First-key ⇒ earliest, last-key ⇒ latest, prefix count ⇒ `bars.len()`; `CoverageRow` shape/`from`/`to`/`bars`/row ordering untouched. Signature stability confirmed: public `coverage` signature and `CoverageRow` unchanged, `coverage_bounds` purely additive, neither the CLI re-export nor the server caller is in the diff.

Panic-free on the coverage path (`coverage_bounds` uses `?` throughout; every `.unwrap()` is `#[cfg(test)]`; clippy `-D warnings` under `unwrap_used=deny` passed). Firewall/scope respected: no `qe-*` edge added, blast radius confined to `crates/storage` (`coverage.rs`, `store.rs`), no Cargo change, no LMDB-schema/persisted-index change (out of scope), no golden/coverage fixture touched.

- nit (non-blocking): `coverage_bounds` counts the whole `(instrument, resolution)` prefix, whereas the old `scan_bars(from_millis(i64::MIN)..from_millis(i64::MAX))` was half-open and excluded a bar at exactly `open_time == i64::MAX`. This is the ONLY semantic difference and unreachable in practice (`i64::MAX` ms ≈ year 292,000,000; no ingest path or fixture produces it) — so "byte-identical on fixtures" holds; the new count-all behaviour is arguably strictly more correct. Design-doc note only.
- nit (non-blocking, pre-existing): `time_from_key` slices `key[n-8..]`, which would underflow on a key shorter than 8 bytes. On the coverage path every iterated key is a bar-prefix key (≥ 9 bytes), and this is the same helper `bar_instruments` already uses — unreachable here, not introduced by this PR.
