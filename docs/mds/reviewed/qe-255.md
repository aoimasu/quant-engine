### Goal
Trigger runs, track status/progress, serve results (ADR D4b file store, D4c subprocess supervision).

### Acceptance criteria (from backlog)
- [x] Creating a run spawns the job, transitions `queued→running→succeeded`, and the result endpoint serves `result.json`.
- [x] A failing job yields `failed` with an error message.

### Implementation summary
- File-based run store `data/runs/<id>/{meta.json,result.json,stdout.log}` + `index.json`; `<id>`=UUID v4
  (opaque). `meta.json` authoritative (status/progress/params/timestamps/error tail), atomic temp+rename writes,
  single-writer-per-run invariant; `index.json` = discovery/order only, serialized by a tokio Mutex.
- `/api` routes: `POST /api/runs` (validate→400 uniform, create, spawn), `GET /api/runs`, `GET /api/runs/:id`,
  `GET /api/runs/:id/result` (404/409 when absent).
- Subprocess supervision (D4c): spawn `qe-cli backtest --run-dir … --json`; concurrent stdout/stderr drain via
  `join!` before `wait()` (no pipe deadlock); stdout JSON-line progress → `meta.json`; success requires
  `done_seen && exit==0` (error-line-then-exit0 ⇒ failed; nonzero ⇒ failed + bounded stderr tail; spawn failure
  ⇒ failed, not 500). Bounded pool = FIFO `tokio::sync::Semaphore(QE_SERVER_MAX_CONCURRENCY, default 2)`, permit
  released on every exit path (success/failure/panic/cancel); `kill_on_drop(true)`.
- `ServerConfig` extended: `data_dir`, `cli_bin` (`QE_SERVER_CLI_BIN` or sibling-of-exe), `max_concurrency` —
  all `QE_SERVER_*`, relative defaults. Injectable `JobSpawner` seam (prod `CliJobSpawner`).
- **deny.toml: NONE** (`uuid` v4 + tokio `process` all MIT/Apache-2.0). Firewall green (qe-cli spawned, not linked).

### DOCUMENTED FOLLOW-UPS (not blockers)
- **No restart reconciliation:** `running` runs orphaned by a server restart are not reconciled in v1 —
  recommend a startup reconciler (candidate follow-up ticket).
- **nit 1 (perf):** blocking `std::fs` I/O on the async runtime + stdout.log reopened per line — fine at admin
  scale; move to `tokio::fs`/`spawn_blocking` if load grows.
- **nit 4 (edge):** a job emitting `done`+exit 0 without writing `result.json` becomes `succeeded` with a 409 on
  `/result` — left as a `// TODO(QE-follow-up)` in `supervise`; consider classifying as failed.
