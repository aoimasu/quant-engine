## Decision summary

New crate `qe-architecture` (zero internal deps, pure `std`). `dependency_graph()` parses each
`crates/*/Cargo.toml` (package name + `qe-*` production deps; section headers classified structurally via a
quote-aware dotted-key parse, so `[dependencies]`, `[dependencies.qe-foo]`, and
`[target.*.dependencies(.qe-foo)]` are all caught; dev-deps excluded). `check_firewall(graph, rules)`
checks the transitive closure against `firewall_rules()`: `qe-wfo ⊥ {qe-ensemble, qe-runtime, qe-venue}`,
`qe-ensemble ⊥ {qe-wfo, qe-runtime, qe-venue}`.

## Review — Approved (passes 1–3)

**Pass 1 → Reviewed (1 blocker).** Design/rules/closure sound; live-injection of a standard `[dependencies]`
edge fails CI. But a concrete **false-negative**: the dependency-**table** form `[dependencies.qe-ensemble]`
(and `[target.*.dependencies]`) — valid production deps cargo recognises — bypassed the exact-string
section match, so a forbidden edge could ship green. Plus a minor (sanity guard checks crate presence, not
that edges parsed) and a note (dev-dep exclusion agreed).

**Pass 2 → Approved (after `15a9197`).** Parser rewritten to classify headers structurally
(`split_dotted_key` is quote-aware; a section is production-deps iff it has a `dependencies`/
`build-dependencies` segment and no `dev-dependencies` segment; a trailing segment names a table dep).
Reviewer re-broke it adversarially — `[target.'cfg(target_os = "linux")'.dependencies.qe-ensemble]`
(nested quotes), `[dependencies.qe-runtime]` + `optional = true`, `[dev-dependencies.qe-ensemble]`
(control) — all classified correctly. Minor #2 fixed (integration test now asserts a known real edge
`qe-runtime → qe-venue`).

**Pass 3 → Approved (after `b0aea0a`).** Applied the pass-2 nit: `classify_section` matches `[package]`
only as the exact single segment, so a `[package.metadata.*].name` can't mis-key the crate. Regression
test added; no other change.

No outstanding comments. Approved & merged.
