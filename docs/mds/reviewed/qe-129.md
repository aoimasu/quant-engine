## Decision summary

New crate `qe-vintage` (Area ⑦). `VintageContent { format_version, vintage_id, chromosomes: Vec<Genome>,
weights: Vec<f64>, calibration: CalibrationProfile, lineage }`; `content_hash` = lowercase-hex SHA-256 over
canonical JSON (same pattern as `Lineage::id`). `Vintage { content, content_hash }`: `seal` validates
(finite + chromosome-aligned weights) then pins the hash; `verify` recomputes; `write`/`load` round-trip
with `load` ALWAYS verifying. `VintageRepository` is the directory-backed store. Downstream of the
search⟂portfolio firewall — bundles both sides' data but stores the ensemble as plain `weights`, so **no
`qe-ensemble` dep**.

## Review (pass 1) — Approved

Principal-level pass against the AC. Verdict: **Approved.**

- **Hash canonicality — the crux — verified.** Grepped `crates/{wfo,risk,determinism}/src` for
  `HashMap`/`HashSet`: **none**. Every hashed field serialises in a fixed order (`VintageContent` +
  `Genome`'s nested structs/arrays in declaration order; `CalibrationProfile`'s `BTreeMap`s key-sorted;
  `Lineage`'s seeds a `Vec`); `f64` weights via serde_json `ryu` (deterministic). No path for two equal
  contents to hash differently; the re-seal test confirms it empirically.
- **(a) Round-trip + hash genuine** — hash covers only `VintageContent` (not the hash field), so
  recompute-and-compare is well-defined; tampering with content *or* the stored hash string is caught.
- **(b) Tamper detection real** — `load` verifies *before* returning (incl. the repository path); mutating
  `weights[0]` → `HashMismatch` from both `verify()` and `load`.
- **(c) Firewall/placement correct** — deps `qe-wfo`/`qe-risk`/`qe-determinism`/serde/sha2/thiserror, **no
  `qe-ensemble`**; a terminal Area-⑦ artefact bundling both sides' data doesn't breach QE-001/QE-132;
  plain `Vec<f64>` weights is the runtime-friendly choice.
- **(d) Error/IO sound; (e) versioning correct (`format_version` hashed); (f) finite-weights documented.**

Two non-blocking notes: (1) hash stability is contractually tied to serde_json formatting (no maps, so
key-ordering is a non-issue; a serde_json major bump could change it) — worth a contract comment; (2) a
non-finite weight serialises to `null` and fails re-load — a seal-time finite check would surface it.

## Review (pass 2, after `90d4879`) — Approved

Both pass-1 notes addressed in code; the AC hash/round-trip path is provably unchanged for valid vintages.

- **(a) Hash path unchanged** — `seal` runs `validate()?` *before* `content_hash()?`; `validate` only
  reads lengths + weight finiteness, never the serialised bytes, so a valid `content()` produces the
  identical digest; all existing tests unaffected.
- **(b) Length-alignment is a sound seal-time invariant** — `weights` is contractually 1:1 with
  `chromosomes`; a mismatch is always an upstream bug. Not over-strict.
- **(c) Both new variants well-formed** (`WeightChromosomeMismatch`, `NonFiniteWeight`) with actionable
  fields; validation order keeps both reachable; the test exercises each (`NaN`→`NonFiniteWeight{index:1}`,
  popped weight→`WeightChromosomeMismatch{weights:1,chromosomes:2}`).
- **(d) No regression** — `validate` is purely additive on the seal path; the hashing-contract change is
  documentation-only.

No outstanding comments. Approved & merged.
