### Goal
Implement the QE-009 kill contract at the venue adapter: flatten-and-halt, independent of cockpit and Hedge
Planner; independently testable trigger.

### Acceptance criteria (from backlog)
- [x] Triggering the kill flattens positions and halts submission even with the cockpit/planner down —
  `kill_flattens_position_and_halts_submission`, `out_of_band_trip_via_cloned_handle_flattens`,
  `gate_honours_kill_switch_conformance`, `flat_first_call_stays_armed_and_still_flattens_a_later_position`.

### Implementation summary
- `VenueKillGate` wraps the QE-217 `VenueSimulator` + a latching `KillHandle`; impls QE-009 `OrderGate`
  (`admit_within_limits=Admit` → default `admit` structurally `FlattenAndHalt` when tripped; passes
  `assert_honours_kill_switch`). `submit()` → `Err(KillHalt)` once tripped; `enforce_kill()` flattens the kept
  position by submitting the closing order from the position alone (no mark/planner needed), latching
  **only after** a non-flat position is actually flattened.
- Closing order sized by `flatten_intent(current_qty)` (opposite side, full magnitude) — mark-free safety
  path. `fill_price` is only the sim execution price, never used to size the flatten.

### Review notes — [Approved] (three passes)

**Pass 1 (@ `540f8024`) — [Reviewed].** Raised F1 (blocker): `enforce_kill` latched `flattened`
unconditionally, so a kill during a keeper-reconnect window (transiently-flat `current_qty`) disarmed the
flatten and could strand an open position on the safety path. Plus F2 (design/code drift on the flatten
sizing) and F3 (duplicated kill-reason fallback between `submit` and `kill_precheck`).

**Pass 2 (@ `571f0adb`) — [Reviewed].** F1 **RESOLVED**: `enforce_kill` now latches only inside the
`Some(intent)` branch, after the closing order is actually submitted; a flat/unknown `current_qty` returns
`Flattened(None)` and stays armed. Reviewer verified both failure modes are avoided — (a) no double-flatten
from keeper fill latency (the latch is set synchronously with the submit, so a later stale qty short-circuits
at the `Halted` guard — exactly one flatten order can ever be sent), and (b) no spurious never-latch (only a
permanently-flat position never latches, which is correct). Regression test
`flat_first_call_stays_armed_and_still_flattens_a_later_position` added. F3 **RESOLVED** via
`OrderGate::kill_reason()` single-source default. F2 partially resolved → opened F4 (residual §D2 drift still
naming `Notional::ZERO`).

**Pass 3 (@ `289a78c1`) — [Approved].** F4 **RESOLVED**: §D2 no longer references `Notional::ZERO`; it now
reads "the flatten is always to flat, computed from the kept position alone via `flatten_intent(current_qty)`
(see D1: no `Notional` target and no mark are needed to size it)", internally consistent with D1 and the code.
Diff since pass 2 was doc-only (design note + work.md), so F1/F3 remained accepted and F2's drift fully closed.
