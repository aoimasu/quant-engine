## Review verdict — [Approved]
**AC pagination stability (id-anchored cursor).** `list_runs_blocking` resolves the cursor by value — `newest_first.iter().position(|e| e.id == cursor).map(|p| p+1)` — so because concurrent creates only prepend to the newest-first view, the cursor id's position shifts but the SET of runs older than it is invariant. `list_runs_paginates_stably_under_concurrent_creates` is non-vacuous: it interleaves a NEW create between page 1 (limit=2) and page 2, then asserts page 2 is exactly [r1, r0] with the new run absent and `next_cursor` null — an offset-based impl would shift and fail this. `limit.unwrap_or(50).clamp(1,200)` (default 50, cap 200, floor 1). Unknown/malformed cursor → `position` None → `start=len` → empty page (no panic, no restart-from-top). Last page ⇒ `next_cursor=None`; `idx<len` at loop exit implies a full page, so `runs.last()` is always `Some` (no dangling cursor).

**AC slim projection + defer.** `RunListItem::project` yields the slim row and defers heavy `params`; a test asserts `row["params"].is_null()` on the list while `GET /api/runs/{id}` still serves `params.is_object()`. Pagination + type/status filter execute at the index level inside the single QE-411 `spawn_blocking` closure — no per-row async, no executor blocking. QE-407 lifecycle intact (`shutdown_drains_running_run_and_stops_accepting` passes).

**AC hook dedup (FE).** `usePollingRun` owns bounded-retry (cap 4, `retrying` flag) + terminal-stop + non-overlapping chained `setTimeout`; BOTH detail screens consume it (the duplicated polling `useEffect`s deleted — BacktestResult's remaining effects are screen-specific: result-view reset + one-shot fetch keyed off `meta.status`; TrainingMonitor has none). `useRunListPolling` owns list refresh with an `inFlight` overlap guard; both lists consume it. Promoted `StatusBadge`/`RunProgress`/`formatRunDate` used in all four screens and exported from `design/index.ts` (copies deleted).

**AC live list refresh.** Lists poll while `anyActive` (queued/running), stop once every row is terminal (vitest asserts the call count freezes after succeeded and no second request for an already-terminal first page — non-vacuous), guard overlaps, and clean up on unmount via `cancelled`+`clearTimeout`.

**Prior work intact.** QE-406 discriminated union preserved; QE-408 `?type=` composes with the new `?status=` (backend test) and the defense-in-depth client `r.type===...` filter is retained on both lists; QE-411 single-`spawn_blocking` batching kept; QE-407 shutdown/reconcile tests pass.

**Wire-shape blast radius.** array→`{runs,next_cursor}` envelope + slim row is breaking, but the only consumer is the SPA (`runs.ts` maps snake_case `next_cursor`→camel `nextCursor`, updated in lockstep) and the backend tests. Grep confirms no CLI/other-Rust/golden reads the list shape (run-protocol's sole hit is a doc comment). `GET /api/runs/{id}` full `RunMeta` unchanged.

**Panic-free / firewall / deps / determinism.** No new unwrap/expect/index-panic on the request path; unknown cursor and empty pages handled; firewall green; no new crate edges; deterministic ordering (append-only index, `iter().rev()`).
