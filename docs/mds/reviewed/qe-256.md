### Goal
Everything must be gated (ADR D4d): Google sign-in restricted to an env allowlist.

### Acceptance criteria (from backlog)
- [x] No session ⇒ 401.
- [x] A valid Google login NOT on the allowlist ⇒ 403.
- [x] An allowlisted login ⇒ 200 and `/api/me` returns the email. (all via a mocked OAuth verifier)

### Implementation summary
- Auth-code flow: `GET /api/auth/login` (v4 `state`, embedded in the Google URL AND a short-lived HttpOnly/Secure/
  SameSite=Lax `qe_oauth_state` cookie) → Google → `GET /api/auth/callback` (double-submit state check; verify ID
  token via injectable `IdTokenVerifier`; `check_claims` enforces `aud`==client_id, `iss`∈Google set, `exp`>now,
  `email_verified`==true; then allowlist).
- Allowlist `QE_ADMIN_ALLOWED_EMAILS` trimmed + case-insensitive, **fail-closed when empty**.
- Signed session cookie `qe_session = b64url(payload).b64url(HMAC-SHA256(payload))`, `payload="v1|<email>|<exp>"`,
  keyed by `QE_SESSION_SECRET`; **constant-time `hmac::verify_slice`** (MAC checked before trusting payload; `exp`
  enforced server-side); `HttpOnly; Secure; SameSite=Lax; Path=/`. `route_layer(require_session)` gates
  `/api/me` + `/api/runs*`; `/api/health` + `/api/auth/*` public. `GET /api/me` → 200 email / 401.
- Verifier seam: sync object-safe `IdTokenVerifier` via `spawn_blocking` (no `async-trait`). Real
  `GoogleOidcVerifier` (ureq + Google **tokeninfo** for signature+expiry) behind default-off `http`; claims still
  re-checked by `check_claims`. `main.rs` wires a `DisabledVerifier` (not accept-all) when `http` off.
- **deny.toml: NONE** — `hmac`/`base64`/`url` (always-on) + `ureq` (behind `http`) added ZERO new Cargo.lock
  crates. `ring`/`rsa`/`jsonwebtoken` deliberately absent (license / RUSTSEC-2023-0071 Marvin). Firewall green.
- QE-255 tests authenticate via `tests/common/mod.rs` using the REAL public `qe_server::mint_session_cookie`
  (production auth not weakened).

### Review — [Approved] (security-critical; 0 blocking, 4 non-blocking nits)
All five make-or-break items verified adversarially: (1) token verification SECURE — independent claim
enforcement, mock-tested; tokeninfo delegation an acceptable documented trade-off, not a hole. (2) CSRF `state`
REAL — browser-bound HttpOnly double-submit cookie; mismatch/missing ⇒ 400. (3) session cookie integrity SECURE
— constant-time MAC, exp server-side, adversarial parse (extra dots / empty MAC / pipe-in-email rejected); no
forged/tampered/wrong-secret cookie validates. (4) gating complete — runs routes merged only inside
`protected_routes()`; no ungated protected route; test-auth uses real signing. (5) fail-closed allowlist +
random ephemeral secret when unset + secrets never logged. (6) firewall/deny/http/async isolation intact.

### DOCUMENTED FOLLOW-UPS (non-blocking nits — candidate hardening tickets)
1. No OIDC `nonce` binding (acceptable for confidential-client code flow) — add nonce for replay defense-in-depth.
2. Local JWKS/RS256 ID-token verification (vs tokeninfo delegation) once a license/advisory-clean verify path
   exists — removes the Google-tokeninfo network dependency.
3. CSRF `state` compared with `==` (fine — random public nonce, not a secret; constant-time not required).
4. `cookie_value` returns the first duplicate cookie (not exploitable); `http`-only `google.rs` path not
   contract-tested (security logic is mock-tested via `check_claims`).
